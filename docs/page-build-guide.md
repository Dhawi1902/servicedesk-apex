# ServiceDesk — 12-Page Build Guide (Oracle APEX 26.1)

**Audience:** the whole team, including non-devs new to APEX.
**Goal:** build all 12 pages of the multi-tenant service desk, page by page, mostly with
declarative (low-code) clicks. Drop to SQL/PL-SQL only where this guide gives you the exact code.

**How to read each page section:** Purpose → Create Page wizard choice → step-by-step build →
region SQL (with the tenant-isolation `WHERE` clause) → buttons/processes/validations →
authorization → APIs used → "Test it" checklist.

**The one rule that cannot break — tenant isolation.** Every client-facing region filters on
`:APP_COMPANY_ID` (and role). The canonical visibility predicate appears on Pages 3, 4, 5 — copy it
verbatim; do not paraphrase it.

> Conventions: DB objects are `UPPER_SNAKE_CASE`. All SQL uses **bind variables** (`:APP_COMPANY_ID`,
> `:P5_TICKET_ID`) — never string-concatenate user input. Page numbers below match §6 of
> `ticketing-system-brief.md`.

## Part A — Before you build any page (prerequisites)

Do these once. Pages 1–12 depend on them.

### A.1 Create the schema (7 tables + ref sequence)

Run in **SQL Workshop → SQL Commands** (or SQL Scripts). This is the canonical column set the rest
of the guide references.

```sql
CREATE TABLE COMPANIES (
  COMPANY_ID    NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  COMPANY_NAME  VARCHAR2(120) NOT NULL,
  COMPANY_TYPE  VARCHAR2(10)  NOT NULL CHECK (COMPANY_TYPE IN ('VENDOR','CLIENT')),
  STATUS        VARCHAR2(10)  DEFAULT 'Active' NOT NULL CHECK (STATUS IN ('Active','Inactive'))
);

CREATE TABLE APP_USERS (
  USER_ID     NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  COMPANY_ID  NUMBER NOT NULL REFERENCES COMPANIES(COMPANY_ID),
  FULL_NAME   VARCHAR2(120) NOT NULL,
  EMAIL       VARCHAR2(255) NOT NULL UNIQUE,
  ROLE        VARCHAR2(20)  NOT NULL
              CHECK (ROLE IN ('Client User','Client Admin','Support Agent','System Admin')),
  STATUS      VARCHAR2(10)  DEFAULT 'Active' NOT NULL CHECK (STATUS IN ('Active','Inactive'))
);

CREATE TABLE CATEGORIES (
  CATEGORY_ID   NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  CATEGORY_NAME VARCHAR2(60) NOT NULL UNIQUE,
  STATUS        VARCHAR2(10) DEFAULT 'Active' NOT NULL
);

CREATE TABLE TICKETS (
  TICKET_ID    NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  TICKET_REF   VARCHAR2(20) UNIQUE,
  COMPANY_ID   NUMBER NOT NULL REFERENCES COMPANIES(COMPANY_ID),   -- the tenant key
  SUBJECT      VARCHAR2(200) NOT NULL,
  DESCRIPTION  CLOB,
  CATEGORY_ID  NUMBER REFERENCES CATEGORIES(CATEGORY_ID),
  PRIORITY     VARCHAR2(10) DEFAULT 'Medium'
               CHECK (PRIORITY IN ('Low','Medium','High','Critical')),
  STATUS       VARCHAR2(15) DEFAULT 'New'
               CHECK (STATUS IN ('New','Assigned','In Progress','On Hold','Resolved','Closed')),
  CREATED_BY   NUMBER REFERENCES APP_USERS(USER_ID),
  ASSIGNED_TO  NUMBER REFERENCES APP_USERS(USER_ID),
  CREATED_AT   TIMESTAMP WITH LOCAL TIME ZONE DEFAULT SYSTIMESTAMP,
  UPDATED_AT   TIMESTAMP WITH LOCAL TIME ZONE DEFAULT SYSTIMESTAMP,
  RESOLVED_AT  TIMESTAMP WITH LOCAL TIME ZONE,
  CLOSED_AT    TIMESTAMP WITH LOCAL TIME ZONE,
  CSAT_SCORE   NUMBER CHECK (CSAT_SCORE BETWEEN 1 AND 5),
  SLA_DUE      TIMESTAMP WITH LOCAL TIME ZONE
);

CREATE TABLE TICKET_COMMENTS (
  COMMENT_ID   NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  TICKET_ID    NUMBER NOT NULL REFERENCES TICKETS(TICKET_ID),
  USER_ID      NUMBER NOT NULL REFERENCES APP_USERS(USER_ID),
  COMMENT_TEXT CLOB NOT NULL,
  IS_INTERNAL  VARCHAR2(1) DEFAULT 'N' NOT NULL CHECK (IS_INTERNAL IN ('Y','N')),
  CREATED_AT   TIMESTAMP WITH LOCAL TIME ZONE DEFAULT SYSTIMESTAMP
);

CREATE TABLE TICKET_HISTORY (
  HISTORY_ID NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  TICKET_ID  NUMBER NOT NULL REFERENCES TICKETS(TICKET_ID),
  USER_ID    NUMBER REFERENCES APP_USERS(USER_ID),
  ACTION     VARCHAR2(100) NOT NULL,
  OLD_VALUE  VARCHAR2(100),
  NEW_VALUE  VARCHAR2(100),
  CREATED_AT TIMESTAMP WITH LOCAL TIME ZONE DEFAULT SYSTIMESTAMP
);

-- Which CLIENT companies (projects) each Support Agent covers.
-- Scopes an agent's queue to their own projects (decision I). A bridge/join table:
-- one agent ↔ many clients, one client ↔ many agents. Composite PK prevents duplicates.
CREATE TABLE AGENT_COMPANIES (
  USER_ID    NUMBER NOT NULL REFERENCES APP_USERS(USER_ID),
  COMPANY_ID NUMBER NOT NULL REFERENCES COMPANIES(COMPANY_ID),
  CONSTRAINT AGENT_COMPANIES_PK PRIMARY KEY (USER_ID, COMPANY_ID)
);

CREATE SEQUENCE TICKET_REF_SEQ START WITH 1 INCREMENT BY 1;

-- Human-friendly TKT-00001 reference, generated on insert
CREATE OR REPLACE TRIGGER TICKETS_BIU_REF
BEFORE INSERT ON TICKETS FOR EACH ROW
WHEN (NEW.TICKET_REF IS NULL)
BEGIN
  :NEW.TICKET_REF := 'TKT-' || LPAD(TICKET_REF_SEQ.NEXTVAL, 5, '0');
END;
/
```

Add a couple of indexes for the tenant key and the queue:
```sql
CREATE INDEX TICKETS_COMPANY_IX  ON TICKETS(COMPANY_ID);
CREATE INDEX TICKETS_ASSIGNED_IX ON TICKETS(ASSIGNED_TO);
CREATE INDEX AGENT_COMPANIES_USER_IX ON AGENT_COMPANIES(USER_ID);
```

Seed sample data from the mockup dataset (`docs/mockups/assets/demo-data.js`): Northwind IT (VENDOR)
+ Acme/Globex/Initech/Umbrella (CLIENT), the 8 users with their roles, 5 categories, and ~12 tickets.
Also seed **`AGENT_COMPANIES`** to match the mockup's `agentCompanies` map — Mike covers Acme + Globex,
Lee covers Acme + Initech (so each agent's queue is scoped to just their projects). You can also
generate volume with **`APEX_DG_DATA_GEN`** (`reference/plsql/025-APEX_DG_DATA_GEN.md`).

### A.2 Create the application

App Builder → **Create** → **New Application** → name `ServiceDesk` → pick a theme (Universal Theme,
default). Add a blank Home page now; we configure the rest below. Set **Features → Configure → Theme

### A.3 The 3 application items

**Shared Components → Application Items → Create** (do this 3 times):

| Name | Scope | Session State Protection |
|---|---|---|
| `APP_COMPANY_ID` | Application | **Restricted: cannot be set from browser** |
| `APP_USER_ID` | Application | **Restricted: cannot be set from browser** |
| `APP_ROLE` | Application | **Restricted: cannot be set from browser** |

The "Restricted" protection matters: it stops a malicious user from passing
`APP_COMPANY_ID` in a URL to see another tenant. These are set **server-side at login only** (A.5).

### A.4 Authentication scheme (login)

**Recommended for a new team: built-in "Application Express Accounts."** APEX stores and verifies
passwords for you — no password hashing to hand-roll. `APP_USERS` then only maps the login (email) to
`ROLE` + `COMPANY_ID`. Create those APEX accounts under **App Builder → … → Manage Users and Groups**,
using each person's email as the username so it matches `APP_USERS.EMAIL`.

Shared Components → **Authentication Schemes → Create → Based on a pre-configured scheme →
Application Express Accounts → Make Current**.

> Alternative (custom table auth): if you must verify against an `APP_USERS.PASSWORD_HASH` column,
> build a **Custom** authentication scheme with an *Authentication Function* that compares
> `STANDARD_HASH(:P9999_PASSWORD || salt, 'SHA256')`. Do **not** use `APEX_UTIL.GET_HASH` for stored
> passwords — by default it salts with *session* info (`p_salted=>TRUE`), so the hash is not
> reproducible across logins (`reference/plsql/061-APEX_UTIL.md` §61.59). Login/session primitives if
> you go fully custom: `APEX_CUSTOM_AUTH.LOGIN` / `POST_LOGIN`
> (`reference/plsql/018-APEX_CUSTOM_AUTH.md`).

### A.5 Post-authentication process (stamp the app items)

On the **Authentication Scheme**, set **Post-Authentication Procedure Name** to a stored procedure, or
paste the body into a Login-page "After Authentication" process. Code:

```plsql
DECLARE
  l_user APP_USERS%ROWTYPE;
BEGIN
  SELECT * INTO l_user
    FROM APP_USERS
   WHERE UPPER(EMAIL) = UPPER(:APP_USER)   -- :APP_USER is the authenticated login
     AND STATUS = 'Active';

  APEX_UTIL.SET_SESSION_STATE('APP_USER_ID',    l_user.USER_ID);
  APEX_UTIL.SET_SESSION_STATE('APP_COMPANY_ID', l_user.COMPANY_ID);
  APEX_UTIL.SET_SESSION_STATE('APP_ROLE',       l_user.ROLE);
EXCEPTION
  WHEN NO_DATA_FOUND THEN
    raise_application_error(-20001, 'Your account is not provisioned. Contact an administrator.');
END;
```
`APEX_UTIL.SET_SESSION_STATE` verified at `reference/plsql/061-APEX_UTIL.md` §61.139.

### A.6 The 4 authorization schemes

Shared Components → **Authorization Schemes → Create** (×4). Type = **Exists SQL Query** (re-checks the
DB each session, so a deactivated/role-changed user loses access immediately). Use these — each returns
true when the query returns ≥1 row:

| Scheme name | Scheme type = "Exists SQL Query" |
|---|---|
| `IS_SYSTEM_ADMIN` | `SELECT 1 FROM APP_USERS WHERE USER_ID = :APP_USER_ID AND ROLE = 'System Admin' AND STATUS='Active'` |
| `IS_AGENT` | `SELECT 1 FROM APP_USERS WHERE USER_ID = :APP_USER_ID AND ROLE = 'Support Agent' AND STATUS='Active'` |
| `IS_CLIENT_ADMIN` | `SELECT 1 FROM APP_USERS WHERE USER_ID = :APP_USER_ID AND ROLE = 'Client Admin' AND STATUS='Active'` |
| `IS_CLIENT_USER` | `SELECT 1 FROM APP_USERS WHERE USER_ID = :APP_USER_ID AND ROLE = 'Client User' AND STATUS='Active'` |

Set **Evaluation Point = Once per session** (fast) — or "Once per page view" if you change roles live in
testing. You can check any scheme in code with `APEX_AUTHORIZATION.IS_AUTHORIZED('IS_SYSTEM_ADMIN')`
(`reference/plsql/011-APEX_AUTHORIZATION.md` §11.3).

> For convenience you may also want a "client side" pseudo-scheme; instead of a 5th scheme, in region
> SQL just test `:APP_ROLE IN ('Client User','Client Admin')`.

### A.7 The canonical tenant-isolation predicate (memorize this)

Every ticket-bearing region uses this exact `WHERE`. It encodes §2 of the brief and the demo's
`visibleTickets()` logic:

```sql
WHERE ( :APP_ROLE = 'System Admin'
     OR ( :APP_ROLE = 'Support Agent' AND ( t.COMPANY_ID IN (SELECT ac.COMPANY_ID FROM AGENT_COMPANIES ac WHERE ac.USER_ID = :APP_USER_ID) OR t.ASSIGNED_TO = :APP_USER_ID ) )
     OR ( :APP_ROLE = 'Client Admin'  AND t.COMPANY_ID  = :APP_COMPANY_ID )
     OR ( :APP_ROLE = 'Client User'   AND t.COMPANY_ID  = :APP_COMPANY_ID AND t.CREATED_BY = :APP_USER_ID ) )
```

> **The agent branch (decision I):** a Support Agent sees tickets for the **client companies they cover**
> — the `t.COMPANY_ID IN (SELECT … FROM AGENT_COMPANIES …)` subquery — plus, defensively, anything
> assigned directly to them. This is what keeps an agent's queue free of clients they're not on. Who
> covers whom is maintained by the System Admin on the **Agent → Project coverage** grid (Page 10a).
> Note this is a vendor-side *usability* filter; the hard client-isolation wall is still the
> `t.COMPANY_ID = :APP_COMPANY_ID` test on the client branches.

## Page 1 — Login

- **Purpose / who / priority:** Email + password sign-in; the post-auth process (A.5) stamps the app
  items. **All roles. MUST.**
- **Page type:** APEX created a **Login Page (page 9999/101)** with the application. Use it as-is.
- **Build steps:**
  1. Confirm **Page → Security → Authentication = Page Is Public** (login must be reachable
     unauthenticated). All other pages stay "Page Requires Authentication" (the default).
  2. The page already has a **Login** region with `P9999_USERNAME`, `P9999_PASSWORD`, and a **Login**
     button that runs the **Login API** process. Leave it.
  3. Branding to match the mockup (`01-login.html`): set the region template title to *ServiceDesk*,
     add the tagline "Multi-tenant support portal" as static content. (Persona quick-login buttons in
     the mockup are demo-only — omit in APEX.)
- **Processes:** the built-in **Login** process calls the authentication scheme → which runs the A.5
  post-auth procedure. Nothing to add.
- **Branch:** built-in "go to application home" after success (we point Home/Dashboard, see Page 2/3).
- **Authorization:** none (public page).
- **APIs:** authentication via the scheme (built-in APEX accounts) or `APEX_CUSTOM_AUTH.LOGIN`
  (`reference/plsql/018-APEX_CUSTOM_AUTH.md` §18.14) if custom. App items set with
  `APEX_UTIL.SET_SESSION_STATE`.
- **Test it:**
  - Valid login lands on Dashboard; `&APP_ROLE.`, `&APP_COMPANY_ID.`, `&APP_USER_ID.` are populated
    (drop them in a temporary Dashboard static region to verify, then remove).
  - Inactive user (set `APP_USERS.STATUS='Inactive'`) is rejected with the provisioning error.
  - Bad password is rejected.

- **Purpose / who / priority:** Routes the user after login; shows a welcome + quick-action cards +

## Page 3 — Dashboard

- **Purpose / who / priority:** Counts by status / priority / company, KPI cards, bar + pie charts,
  role-filtered. **All roles. MUST** (judge non-negotiable). Adds **avg resolution time** and **per-agent
  counts** (SHOULD, FR-28).
- **Page type:** **Blank page**, then add **Cards/Static** KPI regions + **Chart** regions.
- **Build steps:**
  1. Create Page → Blank → name `Dashboard`. Set it as the application's **Home** (Shared Components →
     User Interface Attributes → Home URL = `f?p=&APP_ID.:3:&APP_SESSION.`).
  2. **KPI cards** (4) — one "Static Content / value" region each, or a single Cards region:
     ```sql
     SELECT
       COUNT(*) FILTER (WHERE t.STATUS NOT IN ('Closed','Resolved'))                     open_cnt,
       COUNT(*) FILTER (WHERE t.ASSIGNED_TO IS NULL AND t.STATUS <> 'Closed')            unassigned_cnt,
       COUNT(*) FILTER (WHERE t.STATUS = 'In Progress')                                  inprogress_cnt,
       COUNT(*) FILTER (WHERE t.STATUS IN ('Resolved','Closed'))                         done_cnt
     FROM TICKETS t
     WHERE ( :APP_ROLE = 'System Admin'
          OR ( :APP_ROLE = 'Support Agent' AND ( t.COMPANY_ID IN (SELECT ac.COMPANY_ID FROM AGENT_COMPANIES ac WHERE ac.USER_ID = :APP_USER_ID) OR t.ASSIGNED_TO = :APP_USER_ID ) )
          OR ( :APP_ROLE = 'Client Admin'  AND t.COMPANY_ID  = :APP_COMPANY_ID )
          OR ( :APP_ROLE = 'Client User'   AND t.COMPANY_ID  = :APP_COMPANY_ID AND t.CREATED_BY = :APP_USER_ID ) )
     ```
  3. **Bar chart "Tickets by Status"** — region type **Chart → Bar**, Source SQL:
     ```sql
     SELECT t.STATUS label, COUNT(*) value
       FROM TICKETS t
      WHERE ( :APP_ROLE = 'System Admin'
           OR ( :APP_ROLE = 'Support Agent' AND ( t.COMPANY_ID IN (SELECT ac.COMPANY_ID FROM AGENT_COMPANIES ac WHERE ac.USER_ID = :APP_USER_ID) OR t.ASSIGNED_TO = :APP_USER_ID ) )
           OR ( :APP_ROLE = 'Client Admin'  AND t.COMPANY_ID  = :APP_COMPANY_ID )
           OR ( :APP_ROLE = 'Client User'   AND t.COMPANY_ID  = :APP_COMPANY_ID AND t.CREATED_BY = :APP_USER_ID ) )
      GROUP BY t.STATUS
     ```
  4. **Pie chart "Tickets by Priority"** — region type **Chart → Pie**, same `FROM/WHERE`, `GROUP BY
     t.PRIORITY`, label `t.PRIORITY`.
  5. **"Tickets by Client Company"** classic report — **System Admin only**:
     ```sql
     SELECT c.COMPANY_NAME,
            COUNT(*) FILTER (WHERE t.STATUS NOT IN ('Closed','Resolved')) open_cnt,
            COUNT(*) FILTER (WHERE t.STATUS = 'In Progress')              inprogress_cnt,
            COUNT(*) FILTER (WHERE t.STATUS IN ('Resolved','Closed'))     done_cnt
       FROM COMPANIES c
       LEFT JOIN TICKETS t ON t.COMPANY_ID = c.COMPANY_ID
      WHERE c.COMPANY_TYPE = 'CLIENT'
      GROUP BY c.COMPANY_NAME ORDER BY c.COMPANY_NAME
     ```
     This region is cross-tenant **by design**; gate it with authorization `IS_SYSTEM_ADMIN`.
  6. **Analytics (SHOULD, FR-28)** — KPI "Avg resolution time (hours)" + per-agent bar:
     ```sql
     -- Avg resolution time
     SELECT ROUND(AVG(EXTRACT(DAY FROM (RESOLVED_AT - CREATED_AT))*24
                     + EXTRACT(HOUR FROM (RESOLVED_AT - CREATED_AT))),1) avg_hours
       FROM TICKETS t
      WHERE RESOLVED_AT IS NOT NULL
        AND ( :APP_ROLE = 'System Admin' OR t.COMPANY_ID = :APP_COMPANY_ID );
     -- Per-agent handled (bar chart; show to Agent/Admin)
     SELECT u.FULL_NAME label, COUNT(*) value
       FROM TICKETS t JOIN APP_USERS u ON u.USER_ID = t.ASSIGNED_TO
      WHERE t.STATUS IN ('Resolved','Closed')
      GROUP BY u.FULL_NAME ORDER BY 2 DESC
     ```
- **Authorization:** page-level none. "Tickets by Client Company" region → `IS_SYSTEM_ADMIN`. Per-agent
  region → show when `IS_AGENT` or `IS_SYSTEM_ADMIN` (use a combined scheme or region condition
  `:APP_ROLE IN ('Support Agent','System Admin')`).
- **APIs:** none beyond core SQL. (Export the dashboard with **`APEX_DATA_EXPORT`** if asked —
  `reference/plsql/020-APEX_DATA_EXPORT.md`.)
- **Test it (isolation):**
  - Log in as **Anna (Client User, Acme)** → KPIs count only Anna's own Acme tickets; no company
    breakdown region visible.
  - Log in as **Bob (Client Admin, Acme)** → KPIs cover all Acme tickets, still no other company.
  - Log in as **Sara (System Admin)** → KPIs cover everything; company breakdown region appears.

## Page 4 — Ticket List / Queue

- **Purpose / who / priority:** The main browse / filter / search screen; rows scoped by role + company.
  **All roles. MUST.** (FR-15, FR-16.)
- **Page type:** **Faceted Search** (best for filtering) — or **Interactive Report** if the team
  prefers the familiar IR toolbar. Either works; steps below use Faceted Search.
- **Build steps:**
  1. Create Page → **Faceted Search**, table `TICKETS`. Name `Ticket Queue`.
  2. Replace the region **Source** with this join (gives company name + assignee, applies isolation):
     ```sql
     SELECT t.TICKET_ID, t.TICKET_REF, t.SUBJECT, c.COMPANY_NAME, t.PRIORITY, t.STATUS,
            u.FULL_NAME AS ASSIGNEE, t.UPDATED_AT, t.COMPANY_ID, t.ASSIGNED_TO, t.CREATED_BY
       FROM TICKETS t
       JOIN COMPANIES c       ON c.COMPANY_ID = t.COMPANY_ID
       LEFT JOIN APP_USERS u  ON u.USER_ID    = t.ASSIGNED_TO
      WHERE ( :APP_ROLE = 'System Admin'
           OR ( :APP_ROLE = 'Support Agent' AND ( t.COMPANY_ID IN (SELECT ac.COMPANY_ID FROM AGENT_COMPANIES ac WHERE ac.USER_ID = :APP_USER_ID) OR t.ASSIGNED_TO = :APP_USER_ID ) )
           OR ( :APP_ROLE = 'Client Admin'  AND t.COMPANY_ID  = :APP_COMPANY_ID )
           OR ( :APP_ROLE = 'Client User'   AND t.COMPANY_ID  = :APP_COMPANY_ID AND t.CREATED_BY = :APP_USER_ID ) )
     ```
  3. **Facets** (left panel): add facets on `STATUS`, `PRIORITY`, `CATEGORY_ID` (display category name),
     `ASSIGNED_TO`, and — for admin/agent — `COMPANY_NAME`. Set the `COMPANY_NAME` facet's
     **Server-side Condition** to authorization `IS_SYSTEM_ADMIN` OR `IS_AGENT` so clients never even see
     a company filter.
  4. **Search facet** (keyword): a Search facet across `TICKET_REF, SUBJECT` covers FR-16.
  5. **Columns shown:** Ref (link), Subject, Company (admin/agent only), Priority, Status, Assignee,
     Updated. Make `TICKET_REF` a **link column** → Page 5 with `P5_TICKET_ID = #TICKET_ID#`.
  6. Hide the `COMPANY_NAME` **column** for clients: set its Server-side Condition to
     `:APP_ROLE IN ('System Admin','Support Agent')`. Use Status/Priority **badge** column types for the
     coloured pills in the mockup.
  7. **Self-assign / Grab (row action — Support Agent, decision A).** Add a **link column** "Grab" that
     calls the shared self-assign process for that row. Easiest declarative wiring: a link column whose
     target sets a hidden `P4_TICKET_ID = #TICKET_ID#` and submits a request (e.g. `GRAB`), handled by a
     page process (condition `:REQUEST = 'GRAB'`) that runs
     `SD_ASSIGN_TICKET(:P4_TICKET_ID, :APP_USER_ID, :APP_USER_ID);` then refreshes the region. Show the
     Grab link only on **unowned** rows for **agents**: column Server-side / link condition
     `:APP_ROLE = 'Support Agent' AND '#ASSIGNED_TO#' IS NULL` (or filter in the link SQL on
     `ASSIGNED_TO IS NULL`). See §7a for the process and the "already taken" guard.
- **Buttons:** **New Ticket** (region button) → redirect to Page 6 (modal). Server-side Condition =
  `:APP_ROLE IN ('Client User','Client Admin')`.
- **Authorization:** page-level none (all roles). The page **title** can be dynamic: "My Tickets" for
  clients, "Ticket Queue" for staff — set with a substitution or two regions gated by role.
- **APIs:** none required. `APEX_SEARCH` only if you later add app-wide search.
- **Test it (isolation — the key test):**
  - As **Anna (Client User, Acme)**: only Acme tickets she created; no Globex/Initech rows; no Company
    facet/column.
  - As **Carla (Client Admin, Globex)**: all Globex tickets, zero Acme rows.
  - As **Mike (Support Agent)**: only tickets for **Mike's projects** (Acme + Globex) — zero Initech rows,
    even though Initech has open tickets; Company column visible; the
    **Grab** link shows only on unassigned rows in his projects. Clicking it sets `ASSIGNED_TO`=Mike, status→Assigned,
    writes an `ASSIGN` history row, and the row leaves the "unassigned" set. A second agent no longer sees
    Grab on that row (already taken).
  - As **Sara (System Admin)**: every ticket, every company; no Grab link (admins use the Page 7 modal).
  - **Attack test:** while logged in as Anna, edit the URL to open Page 5 for a Globex ticket id →
    Page 5 must deny it (see Page 5 test).

## Page 5 — Ticket Detail (the hub)

- **Purpose / who / priority:** View/edit one ticket; embeds **Comments** and **History** regions; hosts
  all **lifecycle transition buttons** and **Assign**. Buttons gated by role. **All roles. MUST.**
  (FR-9, FR-11, FR-12, FR-13, FR-27.)
- **Page type:** **Blank page** (not a wizard Form) so you can lay out: a **Form** region on `TICKETS`
  for properties, plus a **Comments** report, plus a **History** timeline.
- **Build steps:**
  1. Create Page → Blank → name `Ticket`. Add page item **`P5_TICKET_ID`** (hidden, **Value Protected =
     On**). The Ticket List link sets it.
  2. **Guard region access** (critical for isolation). Add a **Before Header** PL/SQL process or a
     validation that the current user may see this ticket:
     ```plsql
     DECLARE
       l_ok NUMBER;
     BEGIN
       SELECT COUNT(*) INTO l_ok
         FROM TICKETS t
        WHERE t.TICKET_ID = :P5_TICKET_ID
          AND ( :APP_ROLE = 'System Admin'
             OR ( :APP_ROLE = 'Support Agent' AND ( t.COMPANY_ID IN (SELECT ac.COMPANY_ID FROM AGENT_COMPANIES ac WHERE ac.USER_ID = :APP_USER_ID) OR t.ASSIGNED_TO = :APP_USER_ID ) )
             OR ( :APP_ROLE = 'Client Admin'  AND t.COMPANY_ID  = :APP_COMPANY_ID )
             OR ( :APP_ROLE = 'Client User'   AND t.COMPANY_ID  = :APP_COMPANY_ID AND t.CREATED_BY = :APP_USER_ID ) );
       IF l_ok = 0 THEN
         raise_application_error(-20001, 'You do not have access to this ticket.');
       END IF;
     END;
     ```
     This is the server-side backstop behind the URL — even if someone guesses an id, they are blocked.

### 5a. Lifecycle transition buttons (NOT separate pages)

Add region buttons; each runs a **page process** that updates status **and writes a `TICKET_HISTORY`
row**, then refreshes. Show each button only in the valid from-state and for the allowed role (matches
the demo `transitions()` matrix and §4 of the brief):

| Button | Show when (Server-side Condition) | Sets STATUS → |
|---|---|---|
| Start Work | `:P5_STATUS='Assigned'` AND (`IS_AGENT` or `IS_SYSTEM_ADMIN`) | In Progress |
| Put On Hold | `:P5_STATUS='In Progress'` AND (`IS_AGENT`/`IS_SYSTEM_ADMIN`) | On Hold |
| Resume | `:P5_STATUS='On Hold'` AND (`IS_AGENT`/`IS_SYSTEM_ADMIN`) | In Progress |
| Resolve | `:P5_STATUS='In Progress'` AND (`IS_AGENT`/`IS_SYSTEM_ADMIN`) | Resolved |
| Close | `:P5_STATUS='Resolved'` AND `:APP_ROLE IN ('Client User','Client Admin','System Admin')` | Closed |
| Reopen | `:P5_STATUS='Resolved'` AND `:APP_ROLE IN ('Client User','Client Admin')` | In Progress |
| Escalate (SHOULD, FR-26) | `:P5_STATUS='In Progress'` AND (`IS_AGENT`/`IS_SYSTEM_ADMIN`) | In Progress (+ raise priority + reassign) |

Use **one parameterized process** driven by the button name (set a hidden `P5_NEW_STATUS` via each
button's "Set Items", or branch on `:REQUEST`). Process PL/SQL — **the single place that changes status
and logs history**:

```plsql
DECLARE
  l_old TICKETS.STATUS%TYPE;
BEGIN
  SELECT STATUS INTO l_old FROM TICKETS WHERE TICKET_ID = :P5_TICKET_ID FOR UPDATE;

  UPDATE TICKETS
     SET STATUS      = :P5_NEW_STATUS,
         UPDATED_AT  = SYSTIMESTAMP,
         RESOLVED_AT = CASE WHEN :P5_NEW_STATUS = 'Resolved' THEN SYSTIMESTAMP ELSE RESOLVED_AT END,
         CLOSED_AT   = CASE WHEN :P5_NEW_STATUS = 'Closed'   THEN SYSTIMESTAMP ELSE CLOSED_AT   END
   WHERE TICKET_ID = :P5_TICKET_ID;

  INSERT INTO TICKET_HISTORY (TICKET_ID, USER_ID, ACTION, OLD_VALUE, NEW_VALUE)
  VALUES (:P5_TICKET_ID, :APP_USER_ID,
          CASE :P5_NEW_STATUS
            WHEN 'In Progress' THEN (CASE l_old WHEN 'On Hold' THEN 'Resumed'
                                                WHEN 'Resolved' THEN 'Reopened' ELSE 'Started work' END)
            WHEN 'On Hold'  THEN 'Put on hold'
            WHEN 'Resolved' THEN 'Resolved'
            WHEN 'Closed'   THEN 'Closed'
            ELSE 'Changed status' END,
          l_old, :P5_NEW_STATUS);
END;
```

**Escalate** is a small variant of the same process (FR-26): also `PRIORITY = <next higher>` and
`ASSIGNED_TO = <senior agent>`, and log `ACTION = 'Escalated'`. Keep status `In Progress` — it is an
action, not a new state.

## Add-on — Ticket aging: stale flag & overdue highlight (SHOULD / bonus)

> **Build only after the MUST spine demos with real isolation.** No schema change — pure query + a
> declarative highlight. Lives best as a region on **Page 3 (Dashboard)** or as a variant of the
> **Page 4 (Queue)** report. Grounded against `reference/ui/universal-theme-classes.md` (§9, §12).

### The date-math idiom (use everywhere you show "days")

`*_AT` columns are `TIMESTAMP WITH LOCAL TIME ZONE`. Subtracting two timestamps yields an awkward
`INTERVAL` (day part caps at 99). For whole days, cast both sides to `DATE` first — `CAST(... AS DATE)`
normalizes to the **session** zone, the same zone APEX renders in, and `LOCALTIMESTAMP` is "now" in
that zone:

```sql
FLOOR( CAST(LOCALTIMESTAMP AS DATE) - CAST(last_activity AS DATE) )
```

### Region SQL (one Interactive Report source, drives both features)

`:P3_STALE_DAYS` is a page item holding N (default 3) — the native IR Highlight can't read a page
item, so the configurable threshold must live in SQL.

```sql
SELECT  d.ticket_id, d.ticket_ref, d.subject, d.priority, d.status,
        d.assigned_to_name, d.last_activity, d.days_idle,
        CASE WHEN d.days_idle > :P3_STALE_DAYS THEN 'Y' ELSE 'N' END AS is_stale,
        CASE WHEN d.days_idle > :P3_STALE_DAYS THEN 'st-stale' ELSE 'st-fresh' END AS stale_css,
        CASE
          WHEN d.priority = 'Critical' AND d.days_idle >= 1 THEN 'OVERDUE'
          WHEN d.priority = 'High'     AND d.days_idle >= 2 THEN 'OVERDUE'
          WHEN d.priority = 'Medium'   AND d.days_idle >= 4 THEN 'OVERDUE'
          WHEN d.priority = 'Low'      AND d.days_idle >= 7 THEN 'OVERDUE'
          ELSE 'OK'
        END AS overdue_level,
        CASE
          WHEN d.priority = 'Critical' AND d.days_idle >= 1 THEN 'ov-crit'
          WHEN d.priority = 'High'     AND d.days_idle >= 2 THEN 'ov-high'
          WHEN d.priority = 'Medium'   AND d.days_idle >= 4 THEN 'ov-med'
          WHEN d.priority = 'Low'      AND d.days_idle >= 7 THEN 'ov-low'
          ELSE 'ov-none'
        END AS overdue_css
FROM (
    SELECT a.*, FLOOR( CAST(LOCALTIMESTAMP AS DATE) - CAST(a.last_activity AS DATE) ) AS days_idle
    FROM (
        SELECT  t.ticket_id, t.ticket_ref, t.subject, t.priority, t.status,
                asg.full_name AS assigned_to_name,
                GREATEST(
                    t.updated_at,
                    NVL((SELECT MAX(c.created_at) FROM ticket_comments c WHERE c.ticket_id = t.ticket_id), t.updated_at),
                    NVL((SELECT MAX(h.created_at) FROM ticket_history  h WHERE h.ticket_id = t.ticket_id), t.updated_at)
                ) AS last_activity
        FROM    tickets t
        LEFT JOIN app_users asg ON asg.user_id = t.assigned_to   -- assignee join: NO company filter (see notes)
        WHERE ( :APP_ROLE = 'System Admin'                                                                                    -- §A.7 canonical
             OR ( :APP_ROLE = 'Support Agent' AND ( t.COMPANY_ID IN (SELECT ac.COMPANY_ID FROM AGENT_COMPANIES ac WHERE ac.USER_ID = :APP_USER_ID) OR t.ASSIGNED_TO = :APP_USER_ID ) )
             OR ( :APP_ROLE = 'Client Admin'  AND t.COMPANY_ID  = :APP_COMPANY_ID )
             OR ( :APP_ROLE = 'Client User'   AND t.COMPANY_ID  = :APP_COMPANY_ID AND t.CREATED_BY = :APP_USER_ID ) )       -- role-aware; copy verbatim, do not paraphrase
          AND   t.status NOT IN ('Resolved','Closed')
    ) a
) d
```

- `GREATEST` returns NULL if **any** arg is NULL → each sub-select is wrapped `NVL(..., t.updated_at)`,
  so a ticket with no comments/history still resolves to `updated_at`.
- The priority×idle rule spans **two columns**, which the native IR Highlight can't express — so it's
  collapsed into the single `OVERDUE_LEVEL` flag in SQL, then highlighted off that one column.

### Tenant-isolation notes (read before demoing)

1. **Use the §A.7 canonical role-aware `WHERE` verbatim** — not a bare `company_id = :APP_COMPANY_ID`.
   A plain `company_id` filter would show a **Client User** their whole company's tickets (the role
   matrix scopes them to `CREATED_BY = :APP_USER_ID`) and would blank the Agent/SysAdmin views on this
   shared page. The canonical predicate encodes all four role branches correctly.
2. `TICKET_COMMENTS` / `TICKET_HISTORY` carry **no** `company_id` — the sub-selects are safe **only**
   because they correlate to a `ticket_id` from an already role/tenant-filtered `TICKETS`. Never drop
   the outer predicate, never reuse the sub-selects standalone.
3. **Do NOT** add `:APP_COMPANY_ID` to the `app_users asg` assignee join — agents belong to the
   **vendor** company, not the client tenant; filtering it would blank every assignee name. Tenant
   scope belongs on `TICKETS.company_id` only; the assignee is a safe by-PK lookup. *(Defense-in-depth:
   ensure `TICKETS.ASSIGNED_TO` can only ever hold an agent-role user, so this join can't surface a
   client user's name.)*
4. **Styling is not security.** The highlight/badge runs in the browser on rows already returned.
   `:APP_COMPANY_ID` (and `:APP_ROLE`, `:APP_USER_ID`) must be **Application Items** with Session State
   Protection = *Restricted — may not be set from browser*, stamped server-side at login (§A.5) and
   never exposed as a submittable page item. That — not this query — is the cross-tenant kill-switch.

### Test it (per role — this is the part the isolation audit cares about)
- [ ] **Client User** sees only the open tickets *they raised* — not a colleague's at the same company.
- [ ] **Client Admin** (same company) sees all of that company's open tickets.
- [ ] **Support Agent** sees only open tickets for the client companies they cover (`AGENT_COMPANIES`).
- [ ] **System Admin** sees open tickets across all companies; cross-tenant rows render correctly.
- [ ] A ticket idle > N days shows the stale highlight; add a comment → highlight clears next refresh.
- [ ] A Critical ticket idle ≥ 1 day shows Overdue; a Low one at the same age does not.
- [ ] Assignee name renders (proves note #3 — assignee join not tenant-filtered).
