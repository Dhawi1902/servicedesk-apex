Underpins **judge non-negotiable #2 (multiple companies)** at the root: every tenant that later
shows up isolated in tickets/dashboard/assignment first has to exist as a row here. FR-5 is a
System-Admin-only admin surface (Page 9, Interactive Grid) over the `COMPANIES` table
(`company_id`, `company_name`, `company_type` ∈ `VENDOR`/`CLIENT`, `status` ∈ `ACTIVE`/`INACTIVE`).
Deactivate is a **soft** status flip, not a row delete — `COMPANIES` is referenced by `APP_USERS`,
`TICKETS`, and `AGENT_COMPANIES` FKs with no `ON DELETE CASCADE` (`sql/01_schema.sql` lines 43, 68,
94), so a hard delete of any company with data would fail outright; soft-deactivate is the only
buildable interpretation, not a design shortcut.

## User stories

**Primary (FR-5):** As a **System Admin**, I want to create, edit, and deactivate client companies,
so that the multi-tenant structure the rest of the system depends on (users, tickets, agent
coverage) is under my control and no tenant is ever silently lost or duplicated.

Supporting roles for the negative case: **Client User**, **Client Admin**, **Support Agent** — none
of them manage companies (FR-3 role gating); they only ever *consume* a role-scoped slice of the
company list (never the raw table).

## Acceptance criteria (Given/When/Then)

**AC-1 — Create company (happy path)**
Given I am a System Admin on the Companies grid (Page 9), When I add a row with `company_name`,
`company_type` (`VENDOR`/`CLIENT`), and confirm (`status` defaults `ACTIVE`), Then a new
`COMPANIES` row is inserted and immediately available system-wide — e.g. selectable in the
company picker on Page 10 (Users) the moment I refresh it. There is no separate company list to
keep in sync — every consumer reads `COMPANIES` (or its role-scoped LOV) live.

**AC-2 — Edit company**
Given an existing company row, When I change `company_name` or `company_type` and save, Then the
`COMPANIES` row updates and every dependent surface that reads it live (Page 10 user company
picker, Page 3 dashboard per-company breakdown FR-18, Page 4 ticket list company facet FR-15)
reflects the new value without any code change.

**AC-3 — Deactivate is a status flip, not a delete**
Given an existing company (e.g. seeded Initech), When I set `status = INACTIVE` via the grid, Then
the row is retained as-is (no delete), its `company_id` keeps working for every existing FK
reference (`APP_USERS.company_id`, `TICKETS.company_id`, `AGENT_COMPANIES.company_id`), and the
grid does **not** expose a row-delete action for `COMPANIES` at all — deactivate is the only
removal path (matches the FK design; a hard delete of a referenced row raises an FK violation).

**AC-4 — Negative / page & nav role gate** 🔒
Given I am a Client User, Client Admin, or Support Agent, When I look at the navigation, Then there
is no link to Page 9, and When I attempt to open Page 9 directly by URL, Then access is denied —
gated by the `IS_SYSTEM_ADMIN` authorization scheme (FR-3), not merely a hidden nav entry.

**AC-5 — Negative / write-path gate on the DML process itself** 🔒
Given a non-System-Admin session, When a Companies create/edit/deactivate request is submitted
directly (e.g. a replayed/forged Interactive Grid save call bypassing the UI), Then the write is
rejected server-side because the Interactive Grid's **Automatic Row Processing (DML)** process
itself carries `security.authorizationScheme = IS_SYSTEM_ADMIN` — hiding the page (AC-4) is not
sufficient; the process-level check is the actual gate. Note: unlike the FR-10 write rule,
`COMPANIES` is **not** a `V_MY_*` tenant view and has no row-level `company_id` predicate to
re-check — the sole defense here is the authorization scheme on the DML process, so it must not
be skipped or left only at page level.

**AC-6 — Negative / duplicate company name blocked**
Given an existing company named "Acme Corp", When I try to create or rename another company to
the same name, Then the save is rejected by the `COMPANIES_NAME_UK` unique constraint
(`sql/01_schema.sql` line 26) with a clear on-page error, not a raw ORA exception.

**AC-7 — Non-admin consumers of the company list are role-scoped, never a full cross-tenant list** 🔒
Given a Client User/Admin or a Support Agent uses any surface that shows company information
(Page 4's company filter facet, FR-15; Page 3's per-company breakdown, FR-18), When that surface
renders, Then a Client User/Admin sees **only their own** `company_id` (no facet/breakdown across
companies at all), a Support Agent sees only the companies they cover via `AGENT_COMPANIES`, and
**only System Admin** ever sees the full, unfiltered company list — per `sql/README.md` lines
127-128 ("client-facing user LOVs need `WHERE COMPANY_ID = :APP_COMPANY_ID`; a company picker is
System-Admin-only"). Company **names** are tenant-identifying data; a client seeing another
tenant's company name on a dashboard chart or filter is a leak, not a cosmetic issue.

## Demo path

1. Log in as **sara@northwind.example** (System Admin) → open **Companies** (Page 9) → show the seeded 4 companies (Northwind = VENDOR, Acme/Globex/Initech = CLIENT).
2. **Create:** add a new CLIENT company, e.g. "Umbrella Corp" → row appears in the grid (AC-1).
3. **Edit:** rename it → save → confirm the change (AC-2).
4. **Deactivate:** flip Initech's status to `INACTIVE` → row stays in the grid (not deleted), no delete button ever appears (AC-3).
5. **Downstream check:** open **Users** (Page 10) → the new "Umbrella Corp" is already selectable in the company picker (AC-1/AC-2).
6. **The role-gate punch:** log in as **aaron@acme.example** (Client Admin) → no Companies link in nav, and a direct URL to Page 9 is denied (AC-4); open the dashboard/queue → only Acme ever appears, never Umbrella/Globex/Initech (AC-7).

Reinforces non-negotiable #2 (multiple companies exist and are managed by one role) and
non-negotiable #1 (role-based access — only System Admin touches this page, and every other
role's view of company data is scoped).

## Scope notes (no change needed)

- FR-5 traces cleanly to a committed page (9), committed columns (`COMPANIES.company_type`,
  `status`), and a schema that already enforces "deactivate, don't delete" via FK design (no
  `ON DELETE CASCADE` from `APP_USERS`/`TICKETS`/`AGENT_COMPANIES`). **No brief change required.**
- Adjacent, not blocking: **Page 11 — Categories (manage, SHOULD)** shares the same
  System-Admin-only Interactive Grid pattern as Page 9. Build it after the MUST admin pages (9, 10)
  per §1 priority order — don't let it jump the queue.
- **Parked team decisions, not new requirements** — this page has outsized system-wide blast
  radius and two business rules the brief leaves unspecified. Flag for a quick team call, not a
  silent build choice:
  1. **Deactivate vs. login:** the committed post-auth process (`sql/README.md` step 3) only
     checks `APP_USERS.STATUS`, not `COMPANIES.STATUS` — so today a user at a deactivated company
     can still log in and work within their own (now-inactive) tenant-scoped view. Not a
     tenant-isolation leak (still only their own `company_id`), just unspecified.
  2. **Vendor-row integrity:** nothing currently stops a System Admin from creating a second
     `VENDOR` row, flipping Northwind (the vendor that houses every agent + admin) to `CLIENT`, or
     deactivating the vendor company itself — any of which would orphan the entire support side.
     No AC in this ticket covers it. If the team wants a guard (e.g. block edit/deactivate on the
     row where `company_type = 'VENDOR'`), that's a small brief patch to FR-5's wording, not a new
     FR — raise it before Page 9 is built, since it's cheaper to constrain now than to retrofit.
