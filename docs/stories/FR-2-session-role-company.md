FR-2 is the plumbing under **two of the four judge non-negotiables at once**: role-based access
(#1) and multiple companies (#2) both depend on the app knowing, the instant login succeeds, *who*
the user is, *what role* they hold, and *which company* they belong to. It is not itself a visible
screen — it is the session-state contract (`APP_USER_ID`/`APP_COMPANY_ID`/`APP_ROLE`, stamped by the
post-authentication process from `APP_USERS`) that every other MUST (FR-3, FR-4, FR-9, FR-10, the
dashboard, the queue) reads and trusts. Get this wrong and every downstream isolation/role claim is
built on sand. **This revision corrects one factual error from the initial draft (AC-4's mechanism),
promotes the Session State Protection claim from an assertion to a required live test (AC-8), and
adds three previously-missing failure-mode ACs (AC-7, AC-10, AC-11).**

## User stories

**Primary (FR-2):** As any authenticated user (Client User / Client Admin / Support Agent / System
Admin), I want the system to know my role and company the moment I log in, so that every page I
visit afterward already shows exactly what my role allows, scoped to the right company, with no
extra step from me.

**System Admin distinction:** As a System Admin, I want my stamped company (Northwind, the vendor)
to **not** restrict what I see, so that "my company" context correctly means "all companies" for me
— unlike a Client User, whose stamped company *is* their visibility boundary.

Supporting roles for the negative cases: **Client User**, **Client Admin**, **Support Agent** — each
must get a correctly scoped, non-tamperable session; none can widen their own context.

## Acceptance criteria (Given/When/Then)

**AC-1 — Happy path: successful login stamps all three items**
Given a valid email/password matching an `ACTIVE` row in `APP_USERS`, When authentication succeeds,
Then the post-authentication process runs and sets session state `APP_USER_ID` / `APP_COMPANY_ID` /
`APP_ROLE` to that row's values **before** any application page renders (sql/README
§"Post-Authentication Process"). *Verification during build:* APEX Builder's Session (Debug) →
Session State panel, or a temporary debug region on Page 2. Stamping with no observable consequence
is not "done" — the permanent, judge-visible proof is the role/company-differentiated behavior in
AC-2/AC-3, demoed live (see Demo path).

**AC-2 — Role stamped correctly across all 4 roles (seed check)**
Given each seeded account, When they log in, Then `APP_ROLE` matches exactly:
`sara@northwind.example → SYSTEM_ADMIN`, `mike@northwind.example` / `lena@northwind.example →
SUPPORT_AGENT`, `anna@acme.example` / `george@globex.example` / `ivan@initech.example →
CLIENT_USER`, `aaron@acme.example` / `gina@globex.example → CLIENT_ADMIN`. Observable via Page 2 nav
differing by role (admin-only links appear only for `sara`) — Demo path step 1.

**AC-3 — Company stamped correctly, including the vendor company for staff**
Given the same seeded accounts, When they log in, Then `APP_COMPANY_ID` matches their `APP_USERS`
row: Sara/Mike/Lena → Northwind Support (vendor); Anna/Aaron → Acme Corp; George/Gina → Globex Inc;
Ivan → Initech. Observable via the Queue (Page 4) returning only that company's tickets for a client
role — Demo path step 2.

**AC-4 — Visibility bypass is the view's own role branch, not an authorization scheme** 🔒
*(static-audit)*
Given a Support Agent or System Admin, When any page/process needs to decide what tickets they can
see, Then it must **never** use `APP_COMPANY_ID` as that filter. **Corrected mechanism:**
`V_MY_TICKETS` (`sql/05_isolation_views.sql` L48-49) carries its own independent
`WHEN V('APP_ROLE') = 'SYSTEM_ADMIN' THEN 1` branch, and a Support Agent's scope comes from the
`AGENT_COMPANIES` sub-query in the same view (L58-62) — **neither references any `IS_*`
authorization scheme.** Authorization schemes (`IS_SYSTEM_ADMIN`, `IS_AGENT`, etc.) gate **UI only**
— pages, buttons, columns — never the underlying data filter. `APP_COMPANY_ID` means only "the
company this login's `APP_USERS` row belongs to," never "the companies this user may see." Confirm
by code audit, not by a click.

**AC-5 — Inactive account rejected before any item is set (QA sign-off blocker)**
Given an `APP_USERS` row with `STATUS <> 'ACTIVE'`, When that user attempts to log in, Then
`raise_application_error(-20001, 'Account is not active.')` fires and **no** app item is set — no
partial/stale session state leaks through. **QA blocker, not a footnote:** no seed account currently
has `STATUS <> 'ACTIVE'`, so this path is **not exercisable today** — the test-data owner (P4) must
add one inactive seed user before this AC can be signed off. *Hand-off note:* `apex-expert` must
confirm against `reference/plsql/` (post-authentication procedure semantics) that an exception raised
here actually **aborts session establishment**, rather than logging and continuing — not yet
verified against the reference.

**AC-6 — No matching profile rejected**
Given an APEX Accounts login with no corresponding `APP_USERS` row (email mismatch), When post-auth
runs, Then `NO_DATA_FOUND` → `raise_application_error(-20002, 'No application profile for this
user.')`, and the session ends cleanly rather than proceeding with `NULL` role/company. Same
abort-semantics confirmation as AC-5 applies here.

**AC-7 — Duplicate/case-variant email fails closed, not into an ambiguous session**
Given `APP_USERS.EMAIL` carries a **case-sensitive** `UNIQUE` constraint
(`sql/01_schema.sql` L42, `APP_USERS_EMAIL_UK`) but the post-auth query matches
`WHERE UPPER(EMAIL) = UPPER(:APP_USER)` (`sql/README.md` L73), When two case-variant rows exist for
the same login (e.g. `anna@acme.example` and `Anna@Acme.example`), Then today the `SELECT` returns 2
rows and raises an **unhandled `TOO_MANY_ROWS`** — only `NO_DATA_FOUND` is currently caught.
**Required fix:** add `WHEN TOO_MANY_ROWS THEN raise_application_error(-20003, 'Ambiguous account —
contact System Admin.')` (no partial stamping), **and** fix at the root by making email uniqueness
case-insensitive (functional unique index on `UPPER(EMAIL)`, or normalize to lowercase on insert) so
this state becomes structurally impossible, not just caught.

**AC-8 — App items cannot be overridden from the browser (Session State Protection) — live-clicked
test** 🔒
Given a logged-in session, When a request attempts to set `APP_COMPANY_ID` or `APP_ROLE` via **(a)**
a tampered URL parameter (e.g. an `f?p=` item-value override) and **(b)** a forged/injected hidden
form field on a submit, Then APEX rejects both attempts. This must be an **executed test, clicked
live in the demo** (Demo path step 4) — not a verbal aside. **Prerequisite, verify explicitly, do not
assume:** the application-level **Session State Protection master switch** (Shared Components →
Security Attributes → Session State Protection = Enabled) is ON — without it, the per-item
Restricted setting on `APP_USER_ID`/`APP_COMPANY_ID`/`APP_ROLE` has no effect. This is the **single
point of failure for the entire role/tenant matrix**: `V_MY_TICKETS` branches directly on
`V('APP_ROLE')`/`NV('APP_COMPANY_ID')` with no secondary re-check
(`sql/05_isolation_views.sql` L48-63) — if either were browser-settable, the whole matrix collapses
silently and nothing downstream catches it.

**AC-9 — No page/process re-derives tenant context from a client-editable source (including hidden
mirror items)** 🔒 *(static-audit)*
Given any downstream MUST that needs role/company (FR-3 page gating, FR-4 ticket queries, FR-9
detail, FR-10 assignment, the dashboard), When it reads tenant context, Then it reads
`:APP_COMPANY_ID` / `:APP_ROLE` directly from session state — never from a submittable page item
(**including a developer-created mirror item**, e.g. a convenience `P1_ROLE` or `P4_COMPANY_ID`
that copies the value for display), a URL parameter, or a fresh re-query of `APP_USERS` keyed off
user-supplied input. The mirror-item case is the most likely real-world regression — audit every
page for it, not just the obvious ones.

**AC-10 — FR-2 fails closed, never open** 🔒 *(static-audit + demo manifestation)*
Given the app items are unset (no session, or a session somehow reached outside the post-auth
process), When `V_MY_TICKETS` or any `V_MY_*` view evaluates its `CASE`, Then the `ELSE` branch
returns `0` (`sql/05_isolation_views.sql` L63) — zero rows, **never all rows**. This is FR-2's most
important contract: absence of a stamped role/company must never be read as "show everything."
**Demo manifestation:** hitting any application page URL directly without an authenticated session
redirects to Login (Page 1) before any ticket data renders — there is no unauthenticated path that
reaches a data region.

**AC-11 — Session boundary: concurrent logins never bleed into each other** 🔒 *(code-audit +
two-browser test)*
**Code-audit:** confirm no PL/SQL package-level (global) variable or equivalent cross-session cache
is used anywhere in the build to carry `APP_USER_ID`/`APP_COMPANY_ID`/`APP_ROLE` — APEX application
items are session-scoped by construction, so this guarantee only breaks if someone bypasses the
platform (e.g. a package global "for performance"). **Two-browser test:** log in as
`anna@acme.example` in one browser session and `sara@northwind.example` in another, concurrently;
confirm each session's nav/queue reflects only its own stamped context, with no cross-contamination.

## Demo path (the 30-second "the session already knows" beat)

1. Log in as **sara@northwind.example** (System Admin) → land on **Home/Landing** (Page 2): nav
   shows admin-only links (Companies/Users) — visible proof `APP_ROLE = SYSTEM_ADMIN` was stamped
   (AC-1/AC-2).
2. Log out, log in as **anna@acme.example** (Client User) → nav has no admin links, no assign
   controls; opening the **Queue** (Page 4) shows only Acme Corp tickets — visible proof
   `APP_COMPANY_ID` = Acme and `APP_ROLE = CLIENT_USER` were both stamped correctly (AC-1–AC-3).
3. Log in as **mike@northwind.example** (Support Agent) → Queue is scoped to Acme + Globex via
   `AGENT_COMPANIES`, **not** his own `APP_COMPANY_ID` (Northwind) — the corrected AC-4 point, worth
   calling out explicitly if a judge asks "why doesn't Mike see Northwind's own tickets?"
4. **Live tamper test (clicked, not narrated):** while logged in as Anna, attempt to change role or
   company by editing the URL item value / forging a hidden form field to claim
   `APP_ROLE = SYSTEM_ADMIN` or a different `APP_COMPANY_ID` → APEX rejects it, nav/queue stay
   unchanged as Client User/Acme (AC-8) — the answer to "what stops someone from just editing the
   URL?", now demonstrated live instead of asserted.
5. *(Optional close)* Hit the application URL directly in a fresh, unauthenticated browser tab →
   redirected straight to Login, no data region ever renders (AC-10, fail-closed).
