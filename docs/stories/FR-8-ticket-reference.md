FR-8 isn't one of the judges' four named non-negotiables, but it's the quiet enabler behind a
professional-feeling demo (every ticket shown by a stable `TKT-00001`-style reference, not a raw
numeric ID) and the foundation FR-16 search-by-reference (SHOULD) will need later. It's already
built — `TICKET_REF_SEQ` + the `TICKETS_REF_BI` insert trigger + the `TICKETS_REF_UK` unique
constraint (`sql/01_schema.sql` lines 15, 93, 106–114) — so this ticket pins down the
definition-of-done, closes an update-side immutability gap the schema doesn't yet cover, and defers
the FR-16-dependent isolation checks until that SHOULD is actually built.

## User stories

**Primary (FR-8):** As a **Client User** raising a ticket, I want the system to give my ticket a
short, human-friendly reference (e.g. `TKT-00042`) the moment I submit it, so that I can quote it in
conversation, email, or a later search instead of a meaningless internal ID.

**Supporting:** As a **Support Agent** or **System Admin** working a queue full of tickets across
several companies, I want every ticket to show a stable, unique reference everywhere it appears (list,
detail, history, comments), so that I never confuse two tickets or lose track of one across a session.

## Acceptance criteria (Given/When/Then)

**FR-8's MUST definition-of-done is AC-1–AC-6.** AC-7/AC-8 are genuine isolation concerns but cannot
be tested until FR-16 (search, SHOULD) exists — see their deferred status below.

**AC-1 — Reference auto-generated on create (happy path)**
Given a Client User or Client Admin submits Create Ticket (Page 6, FR-7), When the row inserts into
`TICKETS`, Then the `TICKETS_REF_BI` trigger fires (`WHEN NEW.TICKET_REF IS NULL`) and sets
`TICKET_REF := 'TKT-' || LPAD(TICKET_REF_SEQ.NEXTVAL, 5, '0')` before the row is ever displayed — the
requester never sees a blank or placeholder reference, even for a split second. This applies equally
to a System Admin's "create on behalf of a client" insert path (`sql/05_isolation_views.sql:35-37`).

**AC-2 — Format matches the brief's example**
Given any ticket in the system, When its reference is rendered anywhere, Then it matches
`TKT-` followed by the sequence value zero-padded to a **minimum** of 5 digits (`TKT-00001`,
`TKT-00042`, and — beyond ticket #99,999 — `TKT-100000`, since `LPAD` pads but never truncates). The
5-digit form is the example named in FR-8 (§3 line 140); the format is not a fixed-width regex.

**AC-3 — Reference is globally unique (DB-enforced, not app-enforced)**
Given two tickets are raised in rapid succession — even from **different companies** at effectively
the same instant (e.g. Acme and Initech submit within the same second) — When both rows commit, Then
uniqueness is guaranteed by two independent DB mechanisms: `TICKET_REF_SEQ.NEXTVAL`'s atomic,
session-safe increment (no two callers can ever receive the same value) **and** the `TICKETS_REF_UK`
unique constraint as a hard backstop. Uniqueness is a database guarantee, not something any page logic
could accidentally bypass.

**AC-4 — Reference is not user-settable on any insert path** 🔒
Given the Create Ticket form (Page 6, client-facing) and any System-Admin "create ticket on behalf of
a client" surface, When either is submitted, Then no page item is bound to `TICKET_REF` on **either**
form — there is nothing for a user to set, override, or forge via a tampered POST. The trigger only
assigns a value `WHEN NEW.TICKET_REF IS NULL` (`sql/01_schema.sql:107-113`); a supplied non-null value
is **kept as-is** and bypasses generation entirely, so this rule must hold on every insert surface, not
just Page 6. Flagged for the isolation auditor to confirm no bound `TICKET_REF` item exists anywhere
before demo.

**AC-5 — Reference is immutable after creation (update-side)** 🔒🛠
Given a ticket already has an assigned `TICKET_REF`, When any Interactive Grid or update process on
Pages 4/5/6 saves a row, Then `TICKET_REF` is excluded from Allowed Row DML / never mapped as an
editable item on any of those pages. This matters because `TICKETS_REF_BI` is a `BEFORE INSERT`
trigger **only** — it offers no protection against an `UPDATE` rewriting an existing reference to a
new, still-unique string, and `TICKETS_REF_UK` blocks a *duplicate* rewrite but not a rewrite to an
unused value. Recommended DB-level backstop (build item for apex-expert): add a `BEFORE UPDATE`
trigger on `TICKETS` that raises `raise_application_error` if `:NEW.TICKET_REF != :OLD.TICKET_REF`,
so the guarantee holds even if a future page mistakenly exposes the column. Audit item for the
isolation auditor: confirm no editable `TICKET_REF` surface exists on Pages 4/5/6 before demo.

**AC-6 — Reference displays consistently everywhere the ticket appears**
Given a ticket with reference `TKT-00042`, When it's rendered on the Ticket List/Queue (Page 4), Ticket
Detail (Page 5, incl. as the page title/breadcrumb), and any `TICKET_HISTORY`/comment context that
names the ticket, Then the same `TKT-00042` string is shown in every region — one column, one source
of truth (`TICKETS.ticket_ref`), no per-page reformatting or truncation. Note: Page 5 is keyed/navigated
by `TICKET_ID` (the numeric PK) — `TICKET_REF` there is a read-only display value, never the row-fetch
key. This distinction matters for AC-7/AC-8 below: any future reference-based search must resolve
`TICKET_REF → TICKET_ID` first, then apply the same visibility guard used everywhere else.

**AC-7 — Search by reference respects tenant scope (FR-16, SHOULD)** 🔒 · **Status: deferred
verification, blocked on FR-16 — not part of FR-8's MUST**
Given the Ticket List search box (Page 4, FR-16, not yet built), When a Client User at Company A types
the exact reference of a ticket belonging to Company B, Then the search must return **zero results**
because the search query runs `FROM V_MY_TICKETS` (tenant-scoped view, per `sql/README.md`), never the
base `TICKETS` table, and resolves `TICKET_REF → TICKET_ID` (per AC-6) before applying that guard. A
syntactically valid, real reference for another tenant must be indistinguishable from a non-existent
one. Recorded now so this concern isn't lost; re-audit when FR-16 lands.

**AC-8 — Negative / sequential-reference guessing is not a data-access path** 🔒 · **Status: deferred
verification, blocked on FR-16 — not part of FR-8's MUST**
Given `TICKET_REF` is a predictable, sequential string (`TKT-00001`, `TKT-00002`, …, not a random
token), When a user increments/decrements a reference they know and tries it in search (AC-7, once
built) or in a direct ticket-detail URL/PK, Then access must still be denied by the standard
visibility guard:
```sql
SELECT COUNT(*) INTO l_ok FROM V_MY_TICKETS WHERE TICKET_ID = :P5_TICKET_ID;
IF l_ok = 0 THEN raise_application_error(-20010,'Not authorized'); END IF;
```
(matches `sql/05_isolation_views.sql:31-32` / `sql/README.md:118-121`, the same pattern used in FR-10
AC-6) — the human-friendly reference must never become an easier IDOR path than the numeric
`ticket_id` it sits alongside. Include any FR-16 autocomplete/type-ahead suggestion region in this same
check — a scoped main results report next to an unscoped suggestion LOV still leaks. This is the one
place FR-8's design (readable, sequential) creates a mild new guessability surface versus a random
token, so it's called out explicitly rather than assumed safe by obscurity.

## Demo path (the 20-second "it's real, not a placeholder" beat)

This beat is **self-contained** for FR-8's MUST scope — it does not depend on FR-16 (search) being built.

1. Log in as **anna@acme.example** (Client User) → raise a new ticket (Page 6) → on save, land on
   **Ticket Detail** (Page 5) and point out the fresh `TKT-000NN` reference in the title — generated
   live, not hand-typed (AC-1, AC-2).
2. Log in as **george@globex.example** (Client User, a *different* company) → raise a second ticket
   → its reference is simply the next consecutive number (`TKT-000NN+1`) — no collision, no
   per-company restart (AC-3).
3. Log in as **sara@northwind.example** (System Admin) → open the **Ticket List** (Page 4) → both
   tickets show their references identically, unchanged from creation (AC-5, AC-6).

block the FR-8 demo on it.
