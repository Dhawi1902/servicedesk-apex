FR-11 is the **engine behind the dashboard non-negotiable** (#4): every status count on Page 3
(FR-17/18/19) is a live read of the same `TICKETS.status` this FR mutates, and the history it writes
(FR-12) is what makes the demo's "watch it move through the workflow" beat credible. It also has a
real teeth-to-visibility gap worth calling out up front: Decision I gives an agent **team visibility**
across a covered project's tickets, but only **change authority** on tickets assigned to them or
unassigned — those are two independent checks, and only the first (`V_MY_TICKETS`) is currently
mechanized. The second — an `assigned_to` ownership guard — does not exist anywhere in the committed
SQL yet and is the load-bearing new build item in this FR (AC-6).

## User stories

**Primary (FR-11):** As a **Support Agent**, I want to move a ticket through Assigned → In Progress →
Resolved (and On Hold/Resume if in scope) for tickets assigned to me or unassigned in my covered
projects, so that my work is reflected accurately and the client/dashboard see real progress.

**Supporting (lifecycle table, §4):** As a **Client User** or **Client Admin**, I want to close a
Resolved ticket (confirm it's fixed) or reopen it (it's not), so that my company controls when a
ticket is truly done — using the same status field and history trail as the agent's transitions.

**Supporting negative:** As a **Client User**, I must **not** be able to trigger the agent-side
transitions (Start Work, Hold, Resume, Resolve) — those belong to Support Agent / System Admin only
(FR-3 role gating).

## Acceptance criteria (Given/When/Then)

**AC-1 — Start work (happy path)**
Given a ticket `STATUS = 'Assigned'` and I am the Support Agent it is assigned to (or System Admin),
When I click **Start Work** on Ticket Detail (Page 5), Then the process (a) re-checks role via the
`IS_AGENT`/`IS_SYSTEM_ADMIN` authorization scheme (sql/README L97–100), (b) atomically re-asserts the
from-state — `UPDATE TICKETS SET STATUS = 'In Progress', UPDATED_AT = SYSTIMESTAMP WHERE TICKET_ID =
:P5_TICKET_ID AND STATUS = 'Assigned'` — and (c) inserts a `TICKET_HISTORY` row **only if
`SQL%ROWCOUNT = 1`**, with `ACTION = 'STATUS_CHANGE'` (the fixed enum, `TICKET_HISTORY_ACT_CK`,
01_schema L147–148 — **not** a human-readable verb like `'Started work'`), `OLD_VALUE = 'Assigned'`,
`NEW_VALUE = 'In Progress'`.

**AC-2 — Resolve**
Given a ticket `STATUS = 'In Progress'` and assigned to me (or I am System Admin), When I click
**Resolve**, Then the same atomic-UPDATE + ROWCOUNT-gated-history pattern as AC-1 applies
(`In Progress → Resolved`), `TICKETS.resolved_at` is stamped `SYSTIMESTAMP` in the same `UPDATE`, and
the history row uses `ACTION = 'STATUS_CHANGE'`.

**AC-3 — Close (client confirms, or System Admin) — no extra ownership predicate**
Given a ticket `STATUS = 'Resolved'` and it is **visible to me** per `V_MY_TICKETS` (Client User: I am
`CREATED_BY`; Client Admin: any ticket in my company; System Admin: any ticket — the *existing*
role matrix, no new guard), When I click **Close**, Then status moves `Resolved → Closed`,
`TICKETS.closed_at` is stamped, a `STATUS_CHANGE` history row is written. **Unlike AC-6, do not add an
`AND created_by = :APP_USER_ID` predicate here** — that would incorrectly block a Client Admin
(e.g. `aaron@acme.example`) from closing a ticket a colleague (`anna@acme.example`) raised.
(CSAT capture, FR-27/SHOULD, rides on this transition — adjacent, does not block Close if incomplete;
see Scope notes.)

**AC-4 — On Hold / Resume (contingent on Decision B — currently open, §8)**
Given a ticket `In Progress` and assigned to me (Support Agent) or I am System Admin, When I click
**Put on Hold**, Then status moves `In Progress → On Hold` using the **same** atomic-UPDATE +
ROWCOUNT-gated pattern and the **same AC-6 ownership guard** as Start Work/Resolve — `On Hold`/`Resume`
are not exempt from the ownership check just because they're less central. **These two states are not
yet confirmed for v1** — build only after the minimal spine
(New→Assigned→In Progress→Resolved→Closed, AC-1/2/3/6/7/8/9/10) demos clean, and flag Decision B for
sign-off before committing engineering time here.

**AC-5 — Reopen (contingent on Decision B — currently open, §8)**
Given a ticket `Resolved` and I am the requester's Client User/Client Admin per the AC-3 visibility
rule, When I click **Reopen** (not actually fixed), Then status moves `Resolved → In Progress` (not
back to `New`) with a `STATUS_CHANGE` row where `OLD_VALUE = 'Resolved'`, and **`TICKETS.resolved_at`
is cleared to `NULL`** in the same `UPDATE` (a fresh `Resolve` re-stamps it) — this is required for the
resolution-time/reopen-rate metrics this FR feeds to stay accurate; leaving a stale `resolved_at`
would double-count the first resolution window. Same Decision-B caveat as AC-4.

**AC-6 — Ownership gate: team visibility ≠ change authority (Decision I, Support Agent only)** 🔒
Given a ticket in a company I (Support Agent) cover, but **assigned to a different agent** (e.g. a
Globex ticket assigned to Lena, viewed by Mike who also covers Globex — team visibility per Decision
I), When I attempt any agent-side transition (Start Work, Hold, Resume, Resolve — including via a
tampered ticket PK or forged submit), Then the write is rejected. `V_MY_TICKETS` (05_isolation_views.sql
L45–64) encodes **company-level visibility only** — it has no `assigned_to` predicate — so every
agent-side transition process **must** run this exact check before the state-changing `UPDATE`, not
just AC-9's role check:

```sql
SELECT COUNT(*) INTO l_ok
  FROM V_MY_TICKETS
 WHERE TICKET_ID  = :P5_TICKET_ID
   AND STATUS     = '<expected-from-status>'
   AND ASSIGNED_TO = NV('APP_USER_ID');
IF l_ok = 0 THEN
  raise_application_error(-20010, 'Not authorized');
END IF;
```
This is a **new guard for FR-11** — nothing in the committed SQL enforces it today. Scope the
predicate to `ASSIGNED_TO = NV('APP_USER_ID')` only; an `OR ASSIGNED_TO IS NULL` branch is
unreachable here (a ticket cannot be `Assigned`/`In Progress` while unassigned) — that NULL case is
FR-10's self-assign territory, not FR-11's.

**AC-7 — System Admin is exempt from the ownership guard, never from visibility** 🔒
Given I am System Admin, When I act on any ticket returned by `V_MY_TICKETS` (System Admin sees all —
05_isolation_views.sql L49–50), Then the AC-6 `ASSIGNED_TO = NV('APP_USER_ID')` predicate does **not**
apply to me — I may Start Work/Resolve/Hold any visible ticket regardless of assignee. Implement this
as **two independent `IF` branches**, not one merged condition: (a) the `V_MY_TICKETS` visibility check
runs for **every** role, no exception; (b) the ownership check runs **only** when `V('APP_ROLE') =
'SUPPORT_AGENT'`. Keeping them separate means loosening (b) for System Admin can never accidentally
weaken (a) for anyone else.

**AC-8 — Negative / cross-company status change blocked (all roles)** 🔒
Given a Support Agent who does not cover Company X (e.g. Mike vs Initech) or a Client User from a
different company, When they attempt a status-change action on a ticket outside their scope
(tampered URL/page item), Then the write is rejected by the standard write-rule check
(`SELECT COUNT(*) FROM V_MY_TICKETS WHERE TICKET_ID = :P_ID` → 0 ⇒ `raise_application_error`, per
sql/README L118–121) before any base-table `UPDATE` runs.

**AC-9 — State-machine integrity: no forgeable to-status, no phantom history** 🛠
Given any status-change button submits, When the process runs, Then (a) the **to-status is derived
server-side** from the button/`:REQUEST` value the process itself branches on — **never** read from a
submittable, user-settable page item (`docs/page-build-guide.md` L489/500 currently drives it from
`:P5_NEW_STATUS`, which a forged POST could push directly from `New` to `Closed`); and (b) the
`UPDATE` re-asserts the expected from-status in its `WHERE` clause (AC-1/2 pattern) so a stale or
double-submitted request updates **zero** rows — the `TICKET_HISTORY` insert must be gated on
`SQL%ROWCOUNT = 1`, never run unconditionally. Without both halves, the aging/reopen/FRT dashboard
metrics this FR feeds become corruptible by a single forged or double-clicked request.

**AC-10 — Server-side re-validation, not just hidden buttons** 🔒
Given Ticket Detail renders action buttons per the ticket's current status and my role, When a
transition isn't permitted for my role/state/ownership (e.g. a Client User sees no Start Work
button), Then hiding the button is not sufficient — the underlying process re-checks authorization via
the process-level **Authorization Scheme** (`IS_AGENT`/`IS_SYSTEM_ADMIN`/`IS_CLIENT_USER`/
`IS_CLIENT_ADMIN`, sql/README L91–100) or `APEX_AUTHORIZATION.IS_AUTHORIZED('IS_AGENT')`
(`reference/plsql/011-APEX_AUTHORIZATION.md` §11.3) — not a bare PL/SQL function call — **plus** the
AC-6 ownership guard, so a forged submit is rejected the same as a normal one.

- **`docs/page-build-guide.md` §5a (L473–537) has three defects — do not copy it verbatim:**
  (a) *ACTION-enum violation* — its sample `TICKET_HISTORY` insert (L506–515) writes human-readable
  verbs (`'Started work'`, `'Resumed'`, `'Reopened'`, `'Put on hold'`, `'Changed status'`) into
  `ACTION`, which is constrained to `('STATUS_CHANGE','ASSIGN','ESCALATE','PRIORITY_CHANGE',
  'COMMENT','CSAT')` (`TICKET_HISTORY_ACT_CK`, 01_schema L147–148) — this raises `ORA-02290` at
  runtime; the human-readable verb belongs in a computed display expression, not in `ACTION`.
  (b) *Role/status casing* — its authorization checks (L197–200) and button conditions (L481–486)
  compare against `'Support Agent'`, `'Client User'`, `'Client Admin'`, `'System Admin'`, but the
  committed role model (`APP_USERS_ROLE_CK`, 01_schema L44) and every isolation view/scheme
  (sql/README L91–100, 05_isolation_views.sql) use `'SUPPORT_AGENT'`, `'CLIENT_USER'`,
  `'CLIENT_ADMIN'`, `'SYSTEM_ADMIN'` — every one of the guide's checks matches **zero rows**, silently
  locking out every agent and client. Statuses (`'New'`, `'In Progress'`, etc.) are correctly
  Title-case in both places — don't "fix" those. Resolve in favor of the app-item form
  (`:APP_ROLE`/authorization schemes reading `V('APP_ROLE')`) for consistency with `V_MY_TICKETS`.
  (c) It reads/writes `TICKETS` directly (L497–504) with no `V_MY_TICKETS` visibility gate and no
  ownership guard at all — exactly the AC-6/AC-8 gap this ticket exists to close.
