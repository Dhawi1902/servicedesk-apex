**Judge non-negotiables #1 (role-based access) and #2 (multiple companies)** — FR-7 is
where both get proven for the first time. It's the literal entry point of the ticket
lifecycle (§4: `[*] --> New: Client raises ticket`), gated to Client User/Client Admin
only, and every downstream feature (assignment, history, dashboard counts) inherits
whatever `company_id` gets stamped on this insert. Get the stamp wrong here and the
isolation story is broken before it starts.

## User stories

**Primary (FR-7):** As a **Client User**, I want to raise a support ticket with a
subject, description, category, and priority, so that my company's support team can act
on my issue.

**Secondary (brief §2, Client Admin oversight):** As a **Client Admin**, I want to raise a
ticket the same way a Client User does, so that I can log issues for my team; my extra
oversight (seeing *all* of my company's tickets) happens on the Queue/Dashboard, not on
this form.

Supporting roles for the negative case: **Support Agent** and **System Admin** — per the
lifecycle table (§4), neither triggers "Raise ticket"; they view/manage tickets clients
originate, they don't originate them.

## Acceptance criteria (Given/When/Then)

**AC-1 — Client User happy path**
Given I am logged in as a Client User, When I open **Create/Raise Ticket** (Page 6) from
the queue and submit Subject, Description, Category, and Priority, Then a new `TICKETS`
row is inserted with `status = 'New'`, `company_id = NV('APP_COMPANY_ID')`,
`created_by = NV('APP_USER_ID')`, `created_at` = now, and a unique `ticket_ref` is
generated (FR-8, e.g. `TKT-00009`).

**AC-2 — Client Admin raises the same way**
Given I am logged in as a Client Admin, When I raise a ticket via the same Page 6
process, Then it is created identically (`company_id` = my own company, `created_by` =
me) — Client Admin has no extra field on this form; their oversight (all-company
visibility) is a Page 4/5 concern, not a creation-time one.

**AC-3 — Required-field validation (item-level, not DB-level)**
Given the Create Ticket form, When I submit with Subject, Description, Category, or
Priority left blank, Then the submit is rejected with an inline, field-level error and
no `TICKETS` row is written. Build note 🛠: only `SUBJECT` is DB-level `NOT NULL`
(`01_schema.sql` L79) — `DESCRIPTION` and `CATEGORY_ID` are nullable, and `PRIORITY`
defaults silently to `'Medium'` if left blank (L82). The database will **not** enforce
AC-3 on its own: set `P6_DESCRIPTION`, `P6_CATEGORY`, and `P6_PRIORITY` as page items
with **Required = Yes** and **no item-level default value**, so a blank submit is
caught before the insert, not silently defaulted.

**AC-4 — Tenant stamp is server-side, never client-supplied** 🔒
Given the Create Ticket process, When it inserts the row, Then `company_id` is set from
`NV('APP_COMPANY_ID')` (session state) — **never** read from a submittable page item —
so a tampered form post or URL parameter cannot create a ticket under another company
(sql/README write rule, lines 123–125). App items already carry Session State Protection
= Restricted; no separate SSP acceptance test needed here.

**AC-5 — Negative / role gate, enforced at BOTH the page and the write** 🔒
Given I am a Support Agent or System Admin, When I look for a way to raise a ticket,
Then Page 6 has no nav entry/button (hiding a control is a UX nicety, not the control
itself). The real gate is two enforced layers:
  (a) **Page 6 has a page-level Authorization Scheme** = `IS_CLIENT_USER OR
  IS_CLIENT_ADMIN` (sql/README `IS_*` schemes), so direct navigation to
  `f?p=APP:6:...` is denied; and
  (b) **the insert process itself carries the same authorization condition**
  (re-checkable via `APEX_AUTHORIZATION.IS_AUTHORIZED`), because a process can be
  reached via AJAX/branch independent of whether the page rendered.
Both layers must reject a forged direct submit from a Support Agent or System Admin
session — matches lifecycle §4 (only Client User/Client Admin trigger `— → New`).
Scope note: this is the client-facing create path only; it does not foreclose a
separate System-Admin-only "create on behalf of a client" capability — see Scope notes
below.

**AC-6 — Category LOV is global, priority is a fixed list**
Given the Create Ticket form, When the Category picklist loads, Then it lists all
`CATEGORIES` rows unfiltered by company — categories are shared reference data, not
tenant data (`CATEGORIES` has no `company_id` column, `01_schema.sql` L52–56; sql/README
L128). This is a deliberate absence of a filter, not a leak.

**AC-7 — Output escaping on free text, using the CLOB-safe API where the column is a CLOB** 🛠
Given Subject/Description contain HTML-special characters (e.g. `<script>`), When the
ticket is rendered anywhere off the default Interactive-Report/Form path (Cards on Page
4, an HTML expression on Page 5), Then the text is escaped correctly per column type:
`SUBJECT` (`VARCHAR2(300)`) → `APEX_ESCAPE.HTML`; `DESCRIPTION` (`CLOB`, `01_schema.sql`
L80) → `APEX_ESCAPE.HTML_CLOB` (`APEX_ESCAPE.HTML` is VARCHAR2-only and will error or
truncate against a CLOB). Build note: `sql/README.md` line 130 currently says
`APEX_ESCAPE.HTML` for both `SUBJECT`/`DESCRIPTION`/`COMMENT_TEXT` — that line needs the
same `HTML_CLOB` correction for the two CLOB columns (`DESCRIPTION`, `COMMENT_TEXT`);
see the brief-patch below.

**AC-8 — Unique reference, captured and shown before redirect (FR-8, adjacent MUST)** 🛠
Given a successful create, When the row commits, Then `TICKETS.ticket_ref` is populated
by the existing `TICKETS_REF_BI` before-insert trigger (`TKT-NNNNN`, `01_schema.sql`
L107–113) and is shown to the user immediately (success message and/or the Page 5
redirect). Build note: because the trigger sets `TICKET_REF` **before** insert,
Automatic Row Processing's "Return Key Into" only yields `TICKET_ID` — the create
process must additionally `RETURNING TICKET_REF INTO :P6_TICKET_REF` (or re-select by
the returned `TICKET_ID`) before building the success message/redirect, or AC-8's
"shown immediately" cannot pass.

**AC-9 — Creation is itself a logged history event** 🔒
Given a successful create, When the row commits, Then a `TICKET_HISTORY` row is written
in the **same transaction**: `action = 'STATUS_CHANGE'`, `old_value = NULL`,
`new_value = 'New'`, `user_id = NV('APP_USER_ID')`, timestamp now. Rationale: the brief's
Key rule (§4, "every state change is written to history") and the lifecycle transition
`— → New` both require it; `STATUS_CHANGE` is the correct fixed enum value — there is no
separate `CREATE` action (`TICKET_HISTORY_ACT_CK`, `01_schema.sql` L147–148). This closes
what would otherwise be a silent gap at the very first event of every ticket's activity
log (FR-12). Page 5's History region therefore shows **one creation entry**, not an
empty region, the instant the ticket is opened.

**AC-10 — Immediate visibility, correctly scoped** 🔒
Given a ticket I just created, When I'm redirected to Page 5 (Ticket Detail) or return
to Page 4 (Queue), Then I see it in status `New` with its one creation history entry
(AC-9); When a Client User at a *different* company (e.g. `george@globex.example`) views
their queue, Then my brand-new ticket does **not** appear — `V_MY_TICKETS` scoping
applies identically to freshly committed rows, with no visibility lag or caching gap.
(Assumes Page 4/5 regions and page caching are off — a cached region would give a false
negative on this check, not a real leak.)

**AC-11 — Notification (SHOULD, FR-29 — adjacent, don't block MUST)**
Given a successful ticket creation, When the process commits, Then the requester
receives an automatic acknowledgement email via `APEX_MAIL.SEND`. This is FR-29 (SHOULD)
and rides on the create process — build only after AC-1–AC-10 demo clean. A broken email
must never block or roll back ticket creation.

## Affected pages (of the 12, §6)

- **Page 6 — Create / Raise Ticket** (Form, Modal Dialog, MUST) — primary FR-7 surface;
  stamps `company_id`/`created_by`/`status`, writes the creation `TICKET_HISTORY` row
  (AC-9), triggers FR-8 ref generation + FR-29 ack email. Owner P3.
- **Page 4 — Ticket List/Queue** (MUST) — where Client User/Admin launches Raise Ticket
  from, and where the new ticket first appears (status `New`). Owner P3.
- **Page 5 — Ticket Detail** (MUST) — the ticket lands here immediately after creation
  (post-submit redirect); Comments region starts empty, History region starts with one
  creation entry (AC-9). Owner P2.
- **Page 3 — Dashboard** (MUST) — downstream consumer only: New-status count and
  per-company count (FR-17/18) increment the instant the row commits; not an edit
  surface.

Data touched: `TICKETS` (`ticket_ref`, `company_id`, `subject`, `description`,
`category_id`, `priority`, `status`, `created_by`, `created_at`), `TICKET_HISTORY`
(one `STATUS_CHANGE` row per creation) — all committed columns in sql/README / brief §5.

## Demo path (the 30-second "a ticket is born" beat)

1. Log in as **anna@acme.example** (Client User) → open the **Queue** (Page 4) → click
   **Raise Ticket**.
2. Page 6 modal opens → fill Subject/Description, pick Category + Priority → Submit →
   success message shows the new `TKT-` reference, status **New**; redirected to Page 5
   showing company = **Acme Corp**, created_by = Anna, History with one creation entry
   (AC-9), Comments empty.
3. Back on Page 4, the new ticket sits at the top of Anna's queue.
4. **The isolation punch:** log in as **george@globex.example** (Client User, Globex) →
   open his queue → Anna's brand-new Acme ticket is **absent** (AC-10) — created moments
   ago, still invisible across the tenant boundary.
5. Log in as **sara@northwind.example** (System Admin) → global queue / Dashboard (Page
   3) shows the new ticket under Acme Corp and the New-status count incremented — admin
   sees everything, proving the contrast live.

Hits non-negotiable #2 (multiple companies) hardest, and reinforces #1 (role-based —
only clients could raise it) and #4 (dashboard reacts instantly).
