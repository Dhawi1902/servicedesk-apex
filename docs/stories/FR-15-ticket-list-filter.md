**Judge non-negotiables #1 (role-based) and #2 (multiple companies)** both live in this one screen.
FR-15 is the Ticket List/Queue (Page 4) — the single browse surface for every role, built on
`V_MY_TICKETS` so the row set is *already* tenant/role-scoped before any facet is applied (Decision I
narrows agents further via `AGENT_COMPANIES`). Filtering (status/priority/company/assignee) is UI
convenience over an already-safe row set — the facets must never become a second, weaker access path.

## User stories

**Primary (FR-15):** As a **System Admin**, I want to see a list of every ticket across all client
companies, filterable by status, priority, company, and assignee, so that I can triage the whole
queue and spot unassigned or stuck work at a glance.

**Secondary (FR-15 + Decision I):** As a **Support Agent**, I want to see a list of tickets for
*only the companies I cover*, filterable by status/priority/assignee (and company, limited to my
covered set), so that I can find and pick up work without noise from clients I don't support.

Supporting roles (same page, narrower by design — §6, and FR-15's own text which only promises
*facets* to agents/admins): **Client User** sees only their own tickets; **Client Admin** sees all
of their own company's tickets. Both are negative cases for cross-tenant visibility and for
facet over-exposure below.

## Acceptance criteria (Given/When/Then)

**AC-1 — System Admin sees the full queue (happy path)**
Given I am logged in as System Admin, When I open the Ticket List (Page 4) with no facets applied,
Then I see tickets from **all** companies, sourced from `V_MY_TICKETS` (sql/README "read rule"),
with status, priority, company, and assignee columns visible.

**AC-2 — Support Agent queue is pre-narrowed to covered companies (Decision I)** 🔒
Given I am **mike@northwind.example** (covers Acme + Globex only), When I open Page 4 with no
facets applied, Then I see tickets **only** for Acme and Globex — Initech tickets never appear in
the row set, count, or any facet-combination result (enforced by `V_MY_TICKETS` joining
`AGENT_COMPANIES`, not by a client-side filter).

**AC-3 — Filter by status**
Given any role's row set, When I select a status facet value (e.g. `New`), Then the list shows
only tickets in that status, drawn from the same underlying scoped set (no widening of visibility).

**AC-4 — Filter by priority**
Given any role's row set, When I select a priority facet value (e.g. `High`), Then the list narrows
to that priority within the same tenant/role-scoped set.

**AC-5 — Filter by company (System Admin: all; Agent: covered-only) (Decision I)** 🔒
Given I am System Admin, When the company facet LOV loads, Then it lists **all** companies.
Given I am a Support Agent, When the same facet LOV loads, Then it lists **only** the companies I
cover (my `AGENT_COMPANIES` rows) — Initech never appears in Mike's company facet LOV.

**AC-6 — Filter by assignee, including "Unassigned" (page-scoped LOV, not per-ticket)**
Given the assignee facet, Then its LOV is the **distinct set of `ASSIGNED_TO` values present in
the caller's own `V_MY_TICKETS` row set** (a union across all companies/tickets that caller can
see — e.g. for Mike: every agent who has a ticket in Acme or Globex, himself included), plus a
synthetic **`Unassigned`** value. This is a page-level facet, not FR-10's per-ticket assign-LOV
rule — it must never be built as "agents who cover *this* ticket's company" (that logic doesn't
apply to a facet spanning many tickets).
🛠 **Build note:** `ASSIGNED_TO` (01_schema.sql) is a nullable `NUMBER`; Faceted Search cannot
surface `NULL` as a pickable value, so the facet LOV must use an NVL sentinel
(`NVL(ASSIGNED_TO, -1)` mapped to the label `Unassigned`) for this AC to be testable at all.

**AC-7 — Combined facets narrow, never widen**
Given multiple facets are applied together (e.g. `status=New` + `priority=High` + `company=Globex`),
Then the result is the intersection of all facets **and** the caller's row-visibility scope — a
facet combination can never return a row that `V_MY_TICKETS` alone would have excluded.

**AC-8 — Negative / facet-tampering cannot bypass scope (IDOR via URL/page-item)** 🔒
Given I am a Support Agent who does **not** cover Initech (Mike), When I tamper with the company
facet's URL/page-item value to force `company_id` = Initech's id, Then the list returns **zero
rows** for that value (not Initech's tickets, not an error page) — the guarantee is that the
region SQL is sourced `FROM V_MY_TICKETS` (sql/README "read rule"), so a tampered facet predicate
can only intersect an already-scoped set, never substitute for it. (Note: the three app items
`APP_COMPANY_ID`/`APP_USER_ID`/`APP_ROLE` carry Session State Protection = Restricted per
sql/README "Create 3 Application Items" — that protects *those* items from browser tampering; it
is a separate, additional guarantee from the facet-item behavior asserted here.)

**AC-9 — Negative / Client role has a reduced facet set and no cross-tenant leak** 🔒
Given I am **anna@acme.example** (Client User) or a Client Admin, When I open Page 4, Then I see
only **status** and **priority** facets — no company facet (single-company role) and no assignee
facet (FR-15 promises facets to agents/admins only; exposing the vendor's full agent roster as
filter values to a client is unwanted information disclosure, not a feature). The guarantee is not
UI-absence alone: the region SQL is sourced `FROM V_MY_TICKETS` (fail-closed for clients), so no
facet combination — nor an injected `company_id` on the URL — can ever surface another company's
ticket, ref, or assignee name.

**AC-10 — Self-assign entry point invokes the guarded FR-10 process (Decision A)** 🔒
Given I am a Support Agent viewing an unassigned ticket row in my scoped list, When I use the
row's Self-assign action, Then:
(a) it invokes the **same** shared assign process as FR-10 (`assigned_to = :APP_USER_ID`, status
`New → Assigned`, `TICKET_HISTORY` row) — the list introduces **no second, unguarded**
`UPDATE TICKETS SET ASSIGNED_TO` path of its own;
(b) that process runs the FR-10 write-rule re-check (`SELECT COUNT(*) FROM V_MY_TICKETS WHERE
TICKET_ID = :Pn_TICKET_ID` → 0 ⇒ `raise_application_error`, sql/README "write rule") before the
update commits;
(c) **negative:** if the row-action's ticket PK is tampered to target a ticket in a company Mike
does not cover (e.g. an Initech ticket), the write is rejected by that re-check — mirrors FR-10
AC-6 exactly; hiding the row button is not sufficient.

## Demo path (the 30-second "everyone's queue looks different" beat)

1. Log in as **sara@northwind.example** (System Admin) → open the **Ticket List** (Page 4): full
   queue, all 4 companies visible (AC-1). Filter by status `New`, then add priority `High` — list
   narrows live (AC-3/AC-4/AC-7).
2. Open the **company facet** → all companies listed (AC-5) → pick **Initech** → only Initech
   tickets show.
3. Log in as **mike@northwind.example** (Support Agent) → open Page 4: queue is **pre-narrowed to
   Acme + Globex only**, no Initech row, no Initech option in the company facet (AC-2/AC-5) — **the
   isolation punch**, mirroring FR-10's assign-LOV scoping live.
4. Filter Mike's queue by `assignee = Unassigned` (AC-6) → self-assign one from the row action
   (AC-10) — ties straight into FR-10's demo beat, same guarded process.
5. Log in as **anna@acme.example** (Client User) → open Page 4: only her own Acme tickets, **only
   status/priority facets — no company, no assignee** (AC-9) — confirms the single-company role
   has nothing cross-tenant, and nothing about the agent roster, to leak.

the assignment beat (#3) for FR-10.
