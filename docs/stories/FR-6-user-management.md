**Judge non-negotiables #1 (role-based access) and #2 (multiple companies)** both start here —
FR-6 is the *provisioning* mechanism for both: no user exists with a role or a company until a
System Admin creates one. It's also the quiet dependency behind non-negotiable #3 (assignment):
per Decision I (line 115/433), a Support Agent's queue is scoped by `AGENT_COMPANIES`, so
creating a `SUPPORT_AGENT` user without a coverage row leaves that agent with an empty queue —
FR-10/self-assign silently has nothing to demo. FR-6 must close that loop, not just insert a row.

**Descope note carried through this ticket (protects the demo, §1):** the demoable MUST core of
FR-6 is the `APP_USERS` profile (role + company + status) plus its `AGENT_COMPANIES` coverage.
Provisioning the matching APEX Accounts *login* in-app depends on a workspace-admin privilege
this build's System Admin account does not currently have (AC-2) — that question is split out
as an explicit pre-build spike so a stalled spike never blocks the core demo.

## User stories

**Primary (FR-6):** As a **System Admin**, I want to create, edit, and deactivate users and
assign each one a role and a company, so that every person in the system has the correct access
and every client is correctly tied to their own company.

**Supporting (Decision I):** As a **System Admin**, when I create or edit a user with role
`SUPPORT_AGENT`, I want to also set which client companies that agent covers, so that the agent
actually sees a queue (FR-10 depends on this).

Negative-case roles: **Client User**, **Client Admin**, **Support Agent** — none of them can
reach this page or its actions (FR-3 role gating).

## Acceptance criteria (Given/When/Then)

**AC-1 — Create a user profile (happy path, hard MUST — always demoable)**
Given I am a System Admin on the Users surface (Page 10), When I submit `full_name`, `email`,
`role`, `company_id`, and `status = ACTIVE`, Then a new `APP_USERS` row is inserted. This
acceptance criterion is independent of AC-2 (login provisioning) — the profile write must work
end-to-end even if the login-provisioning spike below is still open.

**AC-2 — Login provisioning is a PRE-BUILD SPIKE, not settled fact** 🛠
`APEX_UTIL.CREATE_USER` requires "the current user must be an APEX workspace administrator and
the application must permit modification of the workspace repository"
(`reference/plsql/061-APEX_UTIL.md` §61.15, line 731). `sql/04_apex_accounts.sql:37`
provisions the business System Admin (`sara@northwind`) as an **END USER**
(`p_developer_privs => NULL`) — her runtime session is *not* a workspace admin, so an in-app
call to `CREATE_USER` (or `EDIT_USER`, same gate, line 1411) fails at demo time as currently
provisioned. Before AC-1's create action is wired to also provision a login, resolve ONE of:
  - **(a) Rejected for this build** — grant the System Admin real workspace-admin dev
    privileges. Over-privilege: blast radius is the whole shared workspace/all apps, a worse
    exposure than a single-company data leak. Do not do this to unblock a demo beat.
  - **(b) Chosen fallback** — the matching APEX Accounts login is provisioned as a **manual,
    out-of-band step** by whoever holds real workspace-admin/DB access (documented runbook,
    not an automated in-app action). `APP_USERS`/role/company stay the only in-app surface.
  - **(c)** Any other mechanism the spike turns up — must be verified against the reference
    before it's asserted as working, not assumed.
No mechanism is asserted as built until this spike closes; AC-9 depends on its outcome.

**AC-3 — Edit a user (role/company change; email is immutable)** 🛠
Given an existing `APP_USERS` row, When I change its `role` and/or `company_id` on the grid,
Then the row updates immediately, but the change only takes effect for that user's **next
login** — the post-auth process re-stamps `APP_COMPANY_ID`/`APP_ROLE` from `APP_USERS` at each
sign-in (sql/README lines 59–86); an already-logged-in session is not live-patched (see AC-11).
`email` is **not editable** after create in this build: it is both the APEX Accounts login
username (set at `CREATE_USER` time, `04_apex_accounts.sql:34`) and the post-auth lookup key
(`WHERE UPPER(EMAIL) = UPPER(:APP_USER)`, README:73); changing it in `APP_USERS` alone without a
matching `EDIT_USER` call (gated by the same workspace-admin restriction as AC-2, line 1411)
silently desyncs the login from the profile and locks the user out. A real email change is a
manual admin recreate, out of scope for the 3-week build.

**AC-4 — Deactivate / reactivate a user**
Given a user row, When I set `status = INACTIVE` and save, Then that person's *next* login
attempt is rejected by the existing post-auth check (`IF l_status <> 'ACTIVE' THEN
raise_application_error(-20001, 'Account is not active.')`, sql/README lines 75–77) — no new
code needed, the guard already exists. Setting `status` back to `ACTIVE` restores login
immediately. This is a soft deactivation (no row delete, no orphaned FKs on
`TICKETS.created_by`/`assigned_to`, `TICKET_COMMENTS.user_id`, `TICKET_HISTORY.user_id`).

**AC-5 — Role must match the assigned company's type (server-side enforced)** 🔒🛠
Given the company picker on the Users form, When I assign role `CLIENT_USER` or `CLIENT_ADMIN`,
Then only companies with `company_type = 'CLIENT'` are valid; When I assign role
`SUPPORT_AGENT` or `SYSTEM_ADMIN`, Then only the `VENDOR` company (Northwind Support) is valid.
The LOV filter is UX only — it must be backed by a **server-side page Validation** (or DB
check/trigger) that re-verifies `role` ↔ `COMPANIES.company_type` on every submit, independent
of what the LOV offered, mirroring the write-rule discipline in sql/README lines 118–121. A
forged IG/form save must not be able to submit a mismatched pair the LOV never showed. This is
the single fastest way to accidentally break tenant isolation at the source (§5, line 231).

**AC-6 — `SUPPORT_AGENT` role requires `AGENT_COMPANIES` coverage, written transactionally
(Decision I)** 🔒🛠
Given I set a user's role to `SUPPORT_AGENT` (create or edit), When I save, Then I must also
pick at least one client company that agent covers, writing row(s) to
`AGENT_COMPANIES(user_id, company_id)` (line 237) **in the same transaction** as the role write
— both commit together or both roll back; a partial failure must never leave an orphaned
coverage row or a `SUPPORT_AGENT` with zero coverage. Without at least one coverage row, that
agent's queue and `V_MY_TICKETS` are correctly — but silently — empty (Decision I, line
115/433). If a user's role is edited *away* from `SUPPORT_AGENT`, their now-orphaned
`AGENT_COMPANIES` rows must be removed in the same transaction, not left dangling (a lingering
row would otherwise keep a demoted ex-agent in FR-10's assignee LOV).

**AC-7 — Negative / role gate, enforced at the page and in the write, not just the button** 🔒🛠
Given I am a Client User, Client Admin, or Support Agent, When I navigate directly to the Users
page URL (Page 10) or attempt to forge a create/edit/deactivate submit, Then access is denied
via **two layers**: (1) a page-level Authorization Scheme `IS_SYSTEM_ADMIN` on Page 10 covering
both the grid's DML process *and* the `AGENT_COMPANIES` sub-region callback, and (2) a
defensive server-side re-check inside the write process itself
(`APEX_AUTHORIZATION.IS_AUTHORIZED('IS_SYSTEM_ADMIN')`,
`reference/plsql/011-APEX_AUTHORIZATION.md` §11.3, line 111). Hiding the button is not
sufficient — an IG Automatic Row Processing replay or an Ajax callback bypasses a show/hide
condition alone.

**AC-8 — Users grid is intentionally unfiltered by `company_id` (by design, not a leak)** 🔒
Given I am the System Admin viewing the Users grid, When it loads, Then it shows **all** users
across **all** companies — this is correct per §2 ("System Admin: sees everything, across all
companies") and is the one page that deliberately does not apply `WHERE company_id =
:APP_COMPANY_ID`. The guard here is role-only (`IS_SYSTEM_ADMIN`, AC-7), not company-scoping —
confirm this distinction explicitly during audit so it isn't flagged as a false-positive leak.

**AC-9 — Provisioning sequence + compensating action (not "rollback")** 🛠
`APEX_UTIL.CREATE_USER` writes the APEX workspace repository, not a plain DML statement on this
schema — it cannot enlist in the same database transaction as an `APP_USERS` insert, so "both
succeed or both roll back" is not atomically implementable. Once AC-2's mechanism is chosen,
the contract is a **sequence with a compensating action**: create the APEX Accounts login
first → then insert `APP_USERS` → if the `APP_USERS` insert fails, delete the just-created login
as compensation; if the login create fails, never attempt the `APP_USERS` insert. The existing
post-auth invariant stays as the safety net for any gap between the two steps (`NO_DATA_FOUND`
→ -20002 "No application profile for this user.", README lines 83–84).

**AC-10 — Duplicate-email create is rejected with a clear message** 🛠
Given a create attempt where the email already exists in `APP_USERS` (or as an APEX Accounts
login — `04_apex_accounts.sql` already "skips users that already exist," README:141), When I
submit, Then the create is rejected before either write with a user-friendly "Email already in
use" message — never a silent no-op and never a raw `ORA-00001` unique-constraint error surfaced
to the System Admin.

**AC-11 — Live-session staleness is an accepted, time-boxed risk (not a gap)**
Given a user is deactivated or has their role/company edited while actively logged in, Then
their already-open session is **not** force-terminated in this build — the change applies at
that user's *next* login only (AC-3, AC-4), mitigated by normal APEX session timeout. This is
documented here as a deliberate, accepted risk for the 3-week timebox. Building session
termination is gold-plating for this demo and is explicitly out of scope.

**AC-12 — Rendered name/email fields are escaped where reused elsewhere** 🔒
Given `full_name`/`email` are rendered on other pages (assignee pickers Page 7, ticket/comment
history attribution), Then those render paths use `APEX_ESCAPE.HTML` (or APEX's default
escaping) on any non-default render path, per the same rule already applied to ticket text
(sql/README line 130).

## Demo path

1. Log in as **sara@northwind.example** (System Admin) → open **Users** (Page 10); show all 8
   seeded users across all 4 companies in one unfiltered grid (AC-8 — correct by design).
2. **Create** a new user profile: `full_name`/`email`, role = `SUPPORT_AGENT`, company =
   Northwind Support (AC-1, AC-5) → save → row appears; a duplicate-email attempt is cleanly
   rejected first (AC-10). Because the role is `SUPPORT_AGENT`, immediately assign coverage to
   **Initech** in the same flow (AC-6 — the Decision I punch: without this step the new agent's
   queue would be empty).
3. **Deactivate** `ivan@initech.example` (status → INACTIVE, AC-4) → attempt to log in as Ivan
   → rejected with "Account is not active." Reactivate → login succeeds again.
4. Log in as **anna@acme.example** (Client User) → try navigating directly to the Users page URL
   → blocked (AC-7, role gate) — reinforces non-negotiable #1 in the same beat.
5. *(Optional closing beat, only if AC-2's spike has resolved before demo day)* show the
   brand-new agent from step 2 signing in live. If the spike is still open, this beat is simply
   cut — steps 1–4 already fully demo non-negotiables #1 and #2 without it (see Scope notes).

Hits non-negotiables #1 (role-based access) and #2 (multiple companies) directly, and sets up
#3 (assignment) by proving a freshly created agent has correctly scoped coverage.
