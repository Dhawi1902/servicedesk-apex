**Judge non-negotiable #1 (role-based access)** — the first thing judges asked for, and the layer everything
else sits on. FR-3 is the **enforcement** layer: authorization schemes gate whole pages, buttons, columns and
regions declaratively, and **every write process re-checks role server-side**. FR-3 is *not* data-scoping —
that is **FR-4** (tenant isolation via `WHERE company_id` / the `V_MY_*` views); this ticket references FR-4
where the two layers meet but does not re-specify it. The load-bearing rule of FR-3: **hiding a control is not
enforcing it.** A hidden Assign button, a suppressed admin page, a masked internal-note column — each must also
reject a forged `f?p` URL or AJAX submit that a real role would never generate.

## User stories

**Page-level (System Admin):** As a **System Admin**, I want the Companies (Page 9) and Users (Page 10)
admin pages to exist only for me, so that no client or agent can reach tenant/user administration — by menu,
by direct URL, or by tampered page number.

**Action-level (Support Agent):** As a **Support Agent**, I want status-change, self-assign and internal-note
controls to appear and function only when my role permits them, so that I can work my tickets without exposing
those actions to clients.

**Column-level (Client User / Client Admin):** As a **Client User** or **Client Admin**, I want to never see
internal agent notes or vendor-only controls on a shared page, so that the client view stays clean and the
vendor's private working notes stay private.

**Enforcement (all roles):** As the **team**, we want every write process to re-verify the caller's role and
ticket visibility server-side, so that a forged submit against a control a role can't see is rejected, not
silently executed.

## Acceptance criteria (Given/When/Then)

**AC-1 — Page-level gate: admin pages are System-Admin-only** 🔒
Given the admin pages Companies (9) and Users (10), When any non-System-Admin (Client User, Client Admin,
Support Agent) requests them, Then the page's **Authorization Scheme = `IS_SYSTEM_ADMIN`** denies access —
both from a missing nav entry **and** from a hand-typed `f?p=…:9:…` URL. Role-specific pages Create Ticket (6,
`IS_CLIENT_USER`/`IS_CLIENT_ADMIN`) and Assign (7, `IS_SYSTEM_ADMIN` + agent self-assign path) follow the same
rule per §6 line 347.

**AC-2 — Action-level gate: buttons appear only for permitted roles**
Given the shared Ticket Detail page (5), When it renders, Then each action button carries the scheme its role
demands — Assign/Reassign = `IS_SYSTEM_ADMIN`; Self-assign = `IS_AGENT` (Decision A); Start work / On Hold /
Resume / Resolve = `IS_AGENT`; Reopen = `IS_CLIENT_USER`/`IS_CLIENT_ADMIN`; Close = client-confirm or
`IS_SYSTEM_ADMIN` (lifecycle §4 lines 204–214). A role that lacks the scheme never sees the button.

**AC-3 — Action-level gate: internal-note toggle is agent-only** 🔒
Given the Add Comment surface (Page 8), When a Client User or Client Admin opens it, Then the **"internal note"
toggle is absent** (gated `IS_AGENT` / `IS_SYSTEM_ADMIN`, FR-14), so a client can never post — or flag — an
internal comment.

**AC-4 — Column-level gate: internal notes hidden from clients** 🔒
Given the comments region on Ticket Detail (5), When a client role views it, Then internal comments
(`is_internal = 'Y'`) do not render. This is already enforced by **`V_MY_COMMENTS`** (sql/README line 108) —
FR-3's column-level requirement is satisfied by selecting `FROM V_MY_COMMENTS`, not by a page-level "hide
column" setting that a client could defeat. *(Data-hiding mechanism is FR-4's view; FR-3 asserts the
requirement.)*

**AC-5 — Enforcement: hidden action still rejects a forged submit** 🔒
Given any write process behind a gated control (assign, self-assign, status change, escalate, comment,
company/user CRUD), When it runs, Then it **re-checks the caller's role server-side** before writing — a forged
`f?p` request or AJAX submit that reaches the process without the required role must
`raise_application_error`, not execute. Hiding the button (AC-2) is necessary but **not sufficient**; the
process is the real gate.

**AC-6 — Enforcement: write-rule ties the role gate to ticket visibility** 🔒
Given any ticket write process, When it fires, Then before mutating the base table it confirms the ticket is
visible to the caller via the committed write-rule
(`SELECT COUNT(*) FROM V_MY_TICKETS WHERE TICKET_ID = :Pn_TICKET_ID` → `0 ⇒ raise_application_error`,
sql/README lines 118–121). This is where FR-3 (role) and FR-4 (tenant/project scope) meet: a Support Agent
acting on a ticket outside their `AGENT_COMPANIES` projects (Decision I) is rejected here even though the
button was visible for their role.

**AC-7 — Negative: role can't be forged from the browser** 🔒
Given `APP_ROLE` / `APP_COMPANY_ID` / `APP_USER_ID` are set once by the post-auth process (sql/README §3),
When a client attempts to elevate by setting `APP_ROLE` via URL or an AJAX call, Then it fails — the app items
are **Session State Protection = Restricted, may not be set from browser** (sql/README §2). Authorization
schemes therefore trust a value the browser cannot forge.

**AC-8 — Positive: one shared page renders correctly per role (live proof)**
Given Ticket List (4), Ticket Detail (5) and Dashboard (3) are built **once** (§6 line 330), When each of the
four roles opens the same page, Then nav, buttons, columns and rows differ by role with **no role-specific page
clones** — proving the authorization-scheme + view design, and giving the judges the "same page, four faces"
beat.

## Demo path (the "same page, four faces" beat)

1. Log in as **sara@northwind.example** (System Admin) → nav shows **Companies (9)** and **Users (10)**; open Ticket Detail (5) — **Assign** visible; internal notes visible (AC-1, AC-2, AC-4).
2. Log in as **mike@northwind.example** (Support Agent) → **no** Companies/Users nav; on a covered ticket, **Self-assign + status buttons** show, internal-note toggle shows (AC-2, AC-3); admin pages blocked by direct URL (AC-1).
3. Log in as **aaron@acme.example** (Client Admin) → sees his company's tickets, but Ticket Detail shows **no Assign, no status buttons, no internal notes**; Add Comment has **no internal toggle** (AC-2, AC-3, AC-4).
4. Log in as **anna@acme.example** (Client User) → same Ticket Detail page, only Raise/Comment/Reopen; **type `f?p=…:10:…`** in the URL bar → **access denied** (AC-1). Attempt a forged assign submit → rejected server-side (AC-5, AC-7).
5. Land the punchline: it was **one** Ticket Detail page (Page 5) the whole time — four roles, four faces, zero clones (AC-8).

Hits non-negotiable #1 (role-based access) head-on and sets up #2/#3/#4 (the gated pages *are* the multi-company, assignment, and dashboard surfaces).
