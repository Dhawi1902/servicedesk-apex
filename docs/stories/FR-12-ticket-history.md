FR-12 has no button of its own — it's the **cross-cutting write-side contract** that every
other lifecycle requirement (FR-10 assign, FR-11 status change, FR-26 escalate, FR-27 CSAT)
must honor. It's the invisible substrate behind two judge non-negotiables: **#3 assignment**
(the demo's "and look, it's logged" moment) and **#4 dashboard** (FR-28's operational
analytics query this table). If any write path skips it, the activity log on Page 5 goes
silent and the audit story — "who did what, when" — collapses without anyone noticing until
a judge asks "how do you know?"

## User stories

**Primary (FR-12):** As a **Client Admin**, I want every status change, assignment, and
escalation on my company's tickets recorded with who-did-it and when, so that I can hold the
vendor accountable and trust the ticket's reported state.

**Operational (FR-9/FR-12):** As a **Support Agent**, I want my own actions (status change,
escalate) automatically logged without extra data entry, so that there's an objective record
if a client disputes what happened and when.

**Oversight:** As a **System Admin**, I want a complete, tamper-proof history across every
company's tickets, so that I can audit agent performance and back the dashboard's operational
metrics (FR-28) with real data.

Supporting role: **Client User** — reads history on their own tickets (FR-9); never writes it
directly (history rows are always a side-effect of another action, never a standalone form).

## Acceptance criteria (Given/When/Then)

**AC-1 — Status change writes history (ties to FR-11, happy path)**
Given a Support Agent moves a ticket `In Progress → Resolved`, When they submit the status
change on Page 5, Then a `TICKET_HISTORY` row is inserted with `action = 'STATUS_CHANGE'`,
`old_value = 'In Progress'`, `new_value = 'Resolved'`, `user_id` = the caller, `created_at` =
now, and `TICKETS.updated_at` is bumped **in the same transaction** — see AC-9 for the
atomicity contract that binds these two writes together.

**AC-2 — Assign/reassign writes history (ties to FR-10)**
Given an admin assign or agent self-assign commits (FR-10 process), When the transaction
completes, Then a `TICKET_HISTORY` row exists with `action = 'ASSIGN'`, `old_value`/`new_value`
holding the prior/new assignee — this is the same guarantee already specified as AC-1/AC-3 in
`FR-10-ticket-assignment.md`; FR-12 is what makes that guarantee non-negotiable everywhere,
not just on the assign path.

**AC-3 — Escalate writes history (ties to FR-26, SHOULD — adjacent, not blocking)**
Given a Support Agent/System Admin escalates an `In Progress` ticket (reassign + raise
priority per §4 line 214–216), When they confirm, Then `TICKET_HISTORY` gains at least one row
capturing the priority `old_value`/`new_value` with `action = 'ESCALATE'`. **Open
implementation question for apex-expert:** whether the reassignment half is folded into that
same `ESCALATE` row or emits a companion `ASSIGN` row — the brief pins the enum (§5 line 246)
but not the row-count shape. Either is compliant with FR-12 as long as **both** the priority
change and the reassignment are individually traceable from the history feed (see AC-9 — this
is why AC-9 cannot require "exactly one" row per action).

**AC-4 — Comment activity on the timeline, correctly scoped for internal notes (ties to FR-13/FR-9/FR-14)** 🔒
Given a user posts a comment, When the Ticket Detail activity feed (Page 5) renders, Then the
comment appears in the same chronological who/what/when timeline as status/assign/escalate
events (FR-9). **Verified against `05_isolation_views.sql`:** `V_MY_HISTORY` (lines 80–83)
filters only on ticket visibility — it carries **no** `is_internal` predicate, while
`V_MY_COMMENTS` (lines 70–75) is the one that hides internal notes from clients
(`IS_INTERNAL = 'N' OR APP_ROLE IN ('SUPPORT_AGENT','SYSTEM_ADMIN')`). Therefore:
- Internal comments (`TICKET_COMMENTS.IS_INTERNAL = 'Y'`) must **never** be mirrored into
  `TICKET_HISTORY` as a `COMMENT` row — doing so would strip the internal-note filter FR-14
  depends on and leak agent-only notes to clients through the audit trail.
- The Page-5 merged timeline must be built as a **UNION of `V_MY_HISTORY` ∪ `V_MY_COMMENTS`**
  (never a bare `TICKET_HISTORY` scan for comment content, never base tables) so the
  `IS_INTERNAL` filter travels with each comment row regardless of which view supplies it.
- The fixed enum's reserved `COMMENT` value (§5 line 246) may still be used if `apex-expert`
  chooses to project comments through history-shaped rows for the UNION — but the *source* of
  truth for visibility is always `V_MY_COMMENTS`, never a mirrored history insert.

**AC-5 — CSAT rating writes history (ties to FR-27, SHOULD — adjacent, not blocking)**
Given a Client User/Admin rates a **Closed** ticket, When they submit the Star Rating, Then
`TICKETS.csat_score` is set **and** a `TICKET_HISTORY` row is written with `action = 'CSAT'`,
`old_value = NULL`, `new_value` = the score.

**AC-6 — History read path is tenant-scoped** 🔒
Given the History region on Page 5, When it queries for rows, Then it selects **only** from
`V_MY_HISTORY` (never the `TICKET_HISTORY` base table) — the read rule already committed in
`sql/README.md` line 112. A Client User at Company A must **never** see a history row
belonging to Company B's ticket, including the identity of the agent who acted on it.

**AC-7 — Negative / IDOR on history read via tampered ticket_id** 🔒
Given a Client User tampers with the ticket-id page item/URL to point at another company's
ticket, When the History region attempts to render, Then `V_MY_HISTORY` returns zero rows
(fail-closed per the view's role/company predicate) — no who/what/when metadata leaks, even
partially (e.g., an agent's name or an escalation event is itself sensitive).

**AC-8 — `user_id` integrity and ticket provenance (non-repudiation)** 🔒
Given any `TICKET_HISTORY` row is written by any process (status change, assign, escalate,
CSAT), When inspected, Then `user_id` is always the authenticated caller
(`NV('APP_USER_ID')`/session state), **never** a value read from a submittable page item —
otherwise a user could forge "who" did an action. `ticket_id`/`company_id` on that row are
trusted only **after** the AC-12 write-guard has passed in the same transaction — non-
repudiation of the actor is moot if the row is allowed to target a foreign ticket. **Footnote:**
`TICKET_HISTORY.USER_ID` has **no `NOT NULL` constraint** at the schema level (`01_schema.sql`
line 139) — this AC is enforced at the PL/SQL layer, not the database; don't let the AC imply
a DB-level guarantee that doesn't exist.

**AC-9 — No lifecycle write commits without a matching history row (the FR-12 contract, cross-cutting, atomic)**
Given any of the write paths in FR-10 (assign), FR-11 (status change), FR-26 (escalate), or
FR-27 (CSAT) runs, Then **at least one** matching `TICKET_HISTORY` row is written for that
action, and none are silently swallowed (AC-3's two-row Escalate shape is compliant — this AC
does not require exactly one). The binding mechanism is **atomicity in both directions**: the
`TICKETS` mutation and its `TICKET_HISTORY` insert commit together or roll back together —
no mutation without a history row, and no orphan history row without the mutation. This is the
actual definition-of-done test to run across **every** lifecycle button before demoing — a
silently-missing history row is a worse failure than a missing feature (brief §1: "a broken
feature scores worse than a missing one") because it looks fine until someone checks the audit
trail on stage. Recommended mechanism (🛠): a single shared PL/SQL history-insert routine
called by every lifecycle process, so this guarantee lives in one place.

**AC-10 — History is append-only / read-only in the UI**
Given any role views the History region, including System Admin, When they look for edit or
delete actions on a history row, Then none exist — the audit trail is immutable from the
application layer (no update/delete process is ever built against `TICKET_HISTORY`).

**AC-11 — Ticket creation is not a history row (scope clarification, not a gap)**
Given a ticket is raised (FR-7, status starts at `New`), When the ticket is created, Then
**no** `TICKET_HISTORY` row is written for the creation itself — `TICKETS.created_by`/
`created_at` already capture that who/when, and the fixed action enum (§5 line 246:
`STATUS_CHANGE`/`ASSIGN`/`ESCALATE`/`PRIORITY_CHANGE`/`COMMENT`/`CSAT`) has no `CREATE` value
because there is no prior state to transition *from*. Don't invent one.

**AC-12 — Negative / write blocked before any mutation or history insert (IDOR on the write side)** 🔒
Given a Client User or Support Agent tampers with `P5_TICKET_ID` (or the URL/PK) to point at a
ticket they cannot see, When **any** lifecycle process runs (status change, assign, escalate,
CSAT), Then the process first re-checks visibility —

```sql
SELECT COUNT(*) INTO l_ok FROM V_MY_TICKETS WHERE TICKET_ID = :P5_TICKET_ID;
IF l_ok = 0 THEN raise_application_error(-20010, 'Not authorized'); END IF;
```

— **before** touching the `TICKETS` base table or inserting into `TICKET_HISTORY` (per
`sql/README.md` lines 115–121, the standing write rule). The foreign ticket must be **neither
mutated nor gain a forged history row**. **Hiding the button is not sufficient** — every
lifecycle process must enforce this server-side, independent of UI state. This is the highest-
severity case in this ticket: an unguarded write path is a cross-tenant *write* plus a forged
audit row, worse than a read leak (AC-7).

## Demo path (the audit-trail proof beat)

1. Log in as **mike@northwind.example** (Support Agent) → open an `Assigned` ticket on
   Page 5 → move it to **In Progress** → point at the **History region**: a new
   `STATUS_CHANGE` line appears immediately with his name and a timestamp (AC-1).
2. Same session → **Escalate** the ticket (if in scope for the build day) → history gains an
   `ESCALATE` line showing the priority bump (AC-3).
3. Log in as **anna@acme.example** (Client User) → open her own ticket → History region shows
   the full timeline (status changes + her own non-internal comments merged, AC-4) — she can
   see it, but only for *her* company's ticket, and never an internal-only note.
4. **The read-isolation punch:** while logged in as Anna, edit the ticket-id in the URL to
   point at a Northwind ticket → History region renders **empty/blocked**, not another
   company's audit trail (AC-6/AC-7).
5. **The write-isolation punch:** still as Anna, replay a captured status-change/assign
   request with the tampered ticket-id → the process rejects it server-side (`-20010`) — the
   foreign ticket is unchanged and gains **no** forged history row (AC-12).
6. Log in as **sara@northwind.example** (System Admin) → open the same ticket → full history
   visible, including Mike's status change and the escalate row — demonstrating the
   System-Admin "everything" view (§2) against the same audit trail.

Reinforces non-negotiable #3 (assignment is logged, not just clicked) and feeds #4 (dashboard
analytics are backed by real, timestamped events, not decoration).
