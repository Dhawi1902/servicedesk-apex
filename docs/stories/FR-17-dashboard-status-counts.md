**Judge non-negotiable #4 (dashboard)** — the visual overview the judges explicitly asked for. FR-17
is the *irreducible* core of that beat: KPI/Card regions on Page 3 counting tickets grouped by
lifecycle status (`New / Assigned / In Progress / On Hold / Resolved / Closed`, §4). The counts are
not decoration — they must be **scoped to the viewer through `V_MY_TICKETS`**, so a wrong number is a
tenant leak, not a cosmetic bug. This ticket is deliberately tight: the status counts, correctly
scoped, on Page 3. Priority/company breakdowns (FR-18), charts (FR-20), and operational analytics
(FR-28) are named as adjacent and kept out of this MUST.

## User stories

**Primary (FR-17):** As a **Support Agent** or **System Admin**, I want the dashboard to show how many
tickets sit in each status, so that I can see the shape of the queue at a glance and decide what to
work next.

**Client-side (FR-17 + FR-19):** As a **Client Admin**, I want the same status counts for **my own
company's** tickets, so that I can see my open workload without seeing anyone else's.

**Client User (FR-17 + FR-19):** As a **Client User**, I want status counts for **my own** tickets, so
that I know how many of my requests are still open versus resolved.

## Acceptance criteria (Given/When/Then)

**AC-1 — One count per lifecycle status**
Given I am on Page 3 (Dashboard), When the region loads, Then I see a distinct count for each of the
six lifecycle statuses — `New`, `Assigned`, `In Progress`, `On Hold`, `Resolved`, `Closed` (§4) —
sourced by grouping `TICKETS.STATUS` from `V_MY_TICKETS`.

**AC-2 — Counts are accurate and sum correctly**
Given the tickets visible to me, When the status counts render, Then each count equals the number of
my visible tickets in that status, and the six counts **sum to my total visible ticket count** (a
total/"All" KPI, if shown, must reconcile).

**AC-3 — Zero-count statuses still render**
Given a status with no visible tickets (e.g. no `On Hold`), When the region loads, Then that status
shows `0` rather than being omitted — the six-slot layout is stable regardless of data (so a fixed
`STATUS` domain drives the grouping, not just statuses that happen to have rows).

**AC-4 — Counts are scoped to the viewer via `V_MY_TICKETS` (isolation)** 🔒
Given I am any role, When the dashboard computes status counts, Then it reads **`FROM V_MY_TICKETS`
only** (never base `TICKETS`), so the numbers reflect exactly the rows I'm permitted to see. The
role-filtering rule itself is **FR-19** — this AC binds FR-17 to that firewall; it does not re-specify
the role matrix.

**AC-5 — Cross-tenant count leak is impossible** 🔒
Given a Client Admin at **Acme** (aaron@acme.example), When they view the dashboard, Then their status
counts include **only** Acme tickets — a Northwind or Initech ticket must never move any Acme count by
one. A count that leaks is a tenant leak (§1 "the one rule that cannot break").

**AC-6 — Client User sees only their own tickets in the counts** 🔒
Given a Client User (anna@acme.example), When the dashboard renders, Then the status counts reflect
**only tickets she raised/owns** — not all of Acme's — consistent with the Client User visibility rule
(FR-19). Aggregation must not become a back door around per-user scoping.

**AC-7 — Counts refresh with the underlying data**
Given a ticket changes status (e.g. `New → Assigned` via FR-10), When I next load or refresh Page 3,
Then the two affected status counts move accordingly — the region reads live from the view, not a
cached snapshot.

## Demo path (the 10-second dashboard beat)

1. Log in as **sara@northwind.example** (System Admin) → land on **Dashboard** (Page 3): status counts
   span **all companies' tickets** across the six statuses (AC-1). This is the judge's "dashboard" hit.
2. Log in as **aaron@acme.example** (Client Admin) → same Dashboard, but every status count now
   reflects **Acme only** — the totals visibly shrink vs. the admin's (AC-5). The isolation punch: the
   numbers themselves are tenant-scoped.
3. Log in as **anna@acme.example** (Client User) → counts shrink again to **just her own** tickets
   (AC-6) — aggregation still respects per-user scope.
4. (Optional) Log in as **mike@northwind.example** (Support Agent) → counts reflect only tickets in the
   companies he covers, same six-status shape.

Hits non-negotiable #4 (dashboard) and re-proves #1 (role-based) and #2 (multi-company) in a single
screen — the counts *are* the isolation demo.
