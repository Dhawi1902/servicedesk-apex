## User stories

**Primary (FR-10):** As a **System Admin**, I want to assign or reassign any ticket to a support
agent, so that work is routed to the right person and the ticket leaves the unassigned queue.

**Self-assign (Decision A):** As a **Support Agent**, I want to self-assign an unassigned ticket
from my project queue, so that I can pick up work without waiting for the admin.

Supporting roles for the negative cases: **Client User** and **Client Admin** — raise/comment/view,
but **no** assign capability (FR-3 role gating).

## Acceptance criteria (Given/When/Then)

**AC-1 — Admin assigns (happy path)**
Given a ticket in status `New` and I am a System Admin, When I open the Assign dialog (Page 7),
pick an agent, and confirm, Then `TICKETS.assigned_to` = that agent, status moves `New → Assigned`,
and a `TICKET_HISTORY` row is written with `action = 'ASSIGN'`, `user_id` = me, timestamp now (FR-12).

**AC-2 — Admin reassigns**
Given a ticket already `Assigned`/`In Progress`, When I pick a different agent, Then `assigned_to`
updates and a new `TICKET_HISTORY` `ASSIGN` row records old→new assignee (`old_value`/`new_value`).
Status is **not** reset by a reassign (only the first New→Assigned transition moves state; §4).

**AC-3 — Agent self-assign (Decision A)**
Given an **unassigned** ticket in a company I cover, and I am a Support Agent, When I click
Self-assign on the queue (Page 4) or detail (Page 5), Then `assigned_to = :APP_USER_ID`, status
`New → Assigned`, and a `TICKET_HISTORY` `ASSIGN` row is written — the *same* process the admin uses.

**AC-4 — Assignee LOV is agents only, scoped to that ticket's company (Decision I)** 🔒
Given the Assign dialog for a ticket owned by Company X, When the agent picklist loads, Then it lists
**only** users whose role = SUPPORT_AGENT and who cover Company X via `AGENT_COMPANIES` — never client
users, never agents who don't cover X. (Seed check: assigning an Initech ticket must **not** offer
Mike; only Lena.)

**AC-5 — Negative / role gate** 🔒
Given I am a Client User or Client Admin, When I view a ticket, Then no Assign button, no Self-assign
button, and the assign process rejects a forged submit — gated by `IS_SYSTEM_ADMIN` (admin) /
`IS_AGENT` (self-assign) (FR-3).

**AC-6 — Negative / cross-project self-assign blocked (isolation)** 🔒
Given a Support Agent who does **not** cover Company X (e.g. Mike vs Initech), When they attempt to
self-assign an X ticket (including via a tampered URL/PK), Then the write is rejected — the process
re-checks visibility (`SELECT COUNT(*) FROM V_MY_TICKETS WHERE TICKET_ID = :P_ID` → 0 ⇒
`raise_application_error`, per sql/README write rule). Hiding the button is not sufficient.

**AC-7 — Negative / cross-tenant assignee leak** 🔒
Given the assignee LOV, When it renders, Then it can never expose users from a company other than the
ticket's own (LOV carries the `company_id`/`AGENT_COMPANIES` predicate, not the base table).

**AC-8 — Notification (SHOULD, FR-21 — adjacent, don't block MUST)**
Given a successful assign, When the process commits, Then the assigned agent is notified via
`APEX_MAIL` (or in-app). This is FR-21 (SHOULD) and rides on the assign process — build only after
AC-1–AC-7 demo clean. A broken email must never take down the assignment demo.

**AC-9 — UI: assignment is visibly reflected** 🎨
Given a ticket is assigned, When the queue/detail re-renders, Then the assignee name and the
`Assigned` status badge are clearly shown (status color chip, assignee avatar/name).

## Demo path (the 30-second assignment beat)

1. Log in as **sara@northwind.example** (System Admin) → open the **Queue** (Page 4); show a `New`, unassigned ticket.
2. Open **Assign** (Page 7) → agent LOV lists only covering agents (AC-4). Assign it → ticket flips to **Assigned**, assignee shown, **History** gains an `ASSIGN` line (Page 5).

5. Log in as **anna@acme.example** (Client User) → open her ticket: **no Assign/Self-assign controls** (AC-5).
