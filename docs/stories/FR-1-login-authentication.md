FR-1 is the **front door to the entire demo** — every judge non-negotiable (role-based access, multiple
companies, assignment, dashboard) is only reachable *after* a successful sign-in, and it is the first thing
a judge does. It is not itself a non-negotiable, but a broken or leaky login takes all four down with it, so
it is treated as critical-path (§7 line 415, P1). Auth route is **APEX Accounts** (built-in salted hashing,
lockout, policy); APEX owns the password, so `APP_USERS` carries **no** password column. The login credential
is the **email itself, matched case-insensitively** (sql/README line 9). The post-login stamping of
`APP_COMPANY_ID`/`APP_USER_ID`/`APP_ROLE` is **FR-2's** concern — referenced here, not duplicated.

## User stories

**Primary (FR-1):** As **any user** (Client User, Client Admin, Support Agent, System Admin), I want to sign
in with my email address and password, so that I reach a session that knows who I am and can gate everything
else I'm allowed to do.

**Negative framing (the gate):** As the **system**, I want to reject anyone without valid credentials and send
unauthenticated visitors back to Login, so that no ticket data is reachable before authentication.

## Acceptance criteria (Given/When/Then)

**AC-1 — Valid sign-in (happy path)**
Given a seeded, active user and the correct password, When I submit the Login page (Page 1) with my email and
password, Then I am authenticated by the APEX Accounts scheme and routed to the Home/Landing page (Page 2,
which may redirect straight to the Dashboard, §6 line 340).

**AC-2 — Email is case-insensitive**
Given the account `anna@acme.example`, When I sign in as `ANNA@ACME.EXAMPLE` (or mixed case) with the correct
password, Then login succeeds identically — APEX Accounts matches the username case-insensitively (sql/README
line 9); no separate account is created or required.

**AC-3 — Wrong password rejected, generic error, no session** 🔒
Given a valid email but an incorrect password, When I submit, Then authentication fails with a **generic**
"Invalid credentials" message (never "wrong password" vs "no such user"), **no** session is established, and
repeated failures trip the APEX Accounts built-in lockout. Hiding nothing — the failure must not reveal
whether the email exists.

**AC-4 — Unknown email rejected identically (no user enumeration)** 🔒
Given an email with no APEX Accounts login, When I submit any password, Then I get the **same** generic
failure and message as AC-3 — a judge (or attacker) cannot distinguish "unknown user" from "wrong password".

**AC-5 — Inactive account cannot sign in**
Given a user whose `APP_USERS.STATUS <> 'ACTIVE'` (e.g. deactivated via FR-6), When correct credentials are
submitted, Then sign-in does not yield a working session — the post-auth process raises and the login is
refused. *Mechanism note:* the status check lives in **FR-2's** post-authentication process
(sql/README lines 75–76); FR-1 owns only the visible outcome (blocked), not that process.

**AC-6 — Unauthenticated access is redirected to Login (no bypass)** 🔒
Given no active session, When I request any application page directly by URL (Dashboard, Ticket Detail, an
admin page), Then APEX redirects me to Page 1 (Login) and no data region renders. The authentication gate is
not optional per page.

**AC-7 — No password material in the tenant table**
Given the committed schema, When `APP_USERS` is inspected, Then it holds **no** password/hash/salt column —
credentials live only in APEX Accounts (sql/README lines 138–140). Login must never read or write a password
on `APP_USERS`.

## Affected pages (of the 12, §6)

- **Page 1 — Login** (Login Page, built-in, MUST) — the only FR-1 surface: email + password entry, submit to
  the Application Express Accounts scheme. Owner P1 (critical path, §7 line 415).
- **Page 2 — Home / Landing** (MUST) — the post-login redirect target (AC-1); routes the authenticated user
  onward. Consumer of FR-1, not an auth surface. Owner P1.
- *Every authenticated page* — inherits the AC-6 gate by being in an app whose authentication scheme is
  Current; not edited here, but each must **not** override page authentication to "Page Is Public".

Data touched: none written by FR-1 itself. Credential store = APEX Accounts (seeded by
`04_apex_accounts.sql`, password `demo`); `APP_USERS` is read (by email) only by FR-2's post-auth process.

## Demo path (the 15-second front-door beat)

1. Land on **Page 1 (Login)**. Sign in as **sara@northwind.example** (System Admin) with `demo` → routed to
   the Dashboard (AC-1). Sign out.
2. Sign in as **anna@acme.example** (Client User) — type it **`ANNA@ACME.EXAMPLE`** to show case-insensitive
   match (AC-2). Sign out.
3. Sign in as **mike@northwind.example** (Support Agent) → reaches his workspace (AC-1). Sign out.
4. **The gate punch:** attempt **mike@northwind.example** with a wrong password → generic "Invalid
   credentials", no entry (AC-3); then a made-up email → *same* message (AC-4, no enumeration).
5. While signed out, paste a deep-link URL (e.g. the Dashboard) → bounced back to Login (AC-6).

Establishes the front door for non-negotiables #1–#4; every later demo beat starts from this login.
