**Judge non-negotiables #1 (role-based access), #2 (multiple companies) and #4 (dashboard)** all land
on this one requirement simultaneously — FR-19 is the rule that the *same* dashboard page must render
**four different truths** depending on who's looking, with zero separate pages (§6, "one page serves
many roles," line 326). Unlike FR-10's negative cases, here **every number on the page is the isolation
surface** — a wrong count *is* a cross-tenant leak, just rendered as an aggregate instead of a row. This
is a cross-cutting rule over FR-17/FR-18's content, not a new query: if every dashboard region reads
`FROM V_MY_TICKETS`, FR-19 is satisfied by construction — the risk is any region that bypasses the view,
any name/label lookup that doesn't, or any drill-down link that leaks a predicate downstream.

## User stories

**Primary (FR-19):** As a **Client Admin**, I want the dashboard to show ticket counts for **only my own
company**, so that I get a true operational overview without seeing (or worrying I'm seeing) another
client's data.

**Contrast (FR-19):** As a **System Admin**, I want the dashboard to show ticket counts **across every
company**, so that I have the global oversight view the role promises.

**Supporting roles for the full matrix:** As a **Client User**, I want the dashboard to reflect only
**my own raised tickets** (narrower than Client Admin's company-wide view — §2 Roles table, line 105).
As a **Support Agent**, I want the dashboard to reflect only tickets in the **companies I cover**
(`AGENT_COMPANIES`, Decision I) — never a company I'm not on.

**Design note (resolves the implicit FR-18 gating question):** the FR-18 "by company" breakdown region
is **not** gated behind `IS_SYSTEM_ADMIN` and is **not** a separate System-Admin-only region. It renders
for every role, sourced from `V_MY_TICKETS` grouped by `company_id` — for a System Admin that naturally
produces a multi-company bar/pie; for a Client Admin or Client User it naturally collapses to a single
bar (their own company); for a Support Agent it shows exactly their covered companies. No extra
visibility logic is needed or wanted — the scoping is a side effect of the view, not a second control.
This is what AC-2/AC-4 below verify.

## Acceptance criteria (Given/When/Then)

**AC-1 — System Admin sees global counts (happy path)**
Given I am logged in as `sara@northwind.example` (System Admin), When I open the Dashboard (Page 3),
Then the status/priority/company breakdowns (FR-17/FR-18) include tickets from **every** seeded
company (Acme, Globex, Initech), with counts matching a raw `SELECT COUNT(*) FROM TICKETS` grouped
the same way.

**AC-2 — Client Admin sees only their own company** 🔒
Given I am logged in as `aaron@acme.example` (Client Admin, Acme Corp), When I open the Dashboard,
Then every count reflects **only Acme tickets** — the total equals
`SELECT COUNT(*) FROM TICKETS WHERE company_id = <Acme>`, Globex and Initech contribute zero, and no
chart category/legend entry for another company appears at all (FR-19 literal text). The company
breakdown region renders as a single Acme bar/slice — not hidden, just naturally scoped (see Design
note above).

**AC-3 — Client User sees only their own tickets (narrower than Client Admin)** 🔒
Given I am logged in as `anna@acme.example` (Client User, Acme Corp), When I open the Dashboard, Then
her count equals `SELECT COUNT(*) FROM TICKETS WHERE company_id = <Acme> AND created_by = <anna>` —
strictly **less than** `aaron@acme.example`'s Acme-wide total from AC-2 — matching the Client User
visibility rule (§2 Roles table, line 105: "Only their own tickets") and the view predicate
`CREATED_BY = NV('APP_USER_ID')` (`05_isolation_views.sql` line 56).

**AC-4 — Support Agent sees only covered companies (Decision I)** 🔒
Given I am logged in as `mike@northwind.example` (Support Agent, covers Acme + Globex), When I open
the Dashboard, Then his total equals
`SELECT COUNT(*) FROM TICKETS WHERE company_id IN (Acme, Globex)` — Initech contributes zero, even
though Mike is vendor-side staff, per `AGENT_COMPANIES` scoping (Decision I). Seed check: Lena, who
covers Globex + Initech, sees the inverse split — her total includes Globex + Initech, **never** Acme.
The company breakdown region shows exactly two bars/slices for Mike, never a third.

**AC-5 — Fail-closed on missing/unrecognized role**
Given a session where `APP_ROLE` is unset, blank, or not one of the four recognized values, When the
Dashboard loads, Then every region shows **zero** (the view's `ELSE 0` branch,
`05_isolation_views.sql` line 63–64) — never all data, and never an unhandled error page. Fail-closed,
not fail-open.

**AC-6 — Negative / session-state tamper via long-form URL syntax** 🔒
Given a Client Admin or Client User session, When the long-form URL item-setting syntax
(`f?p=app:3:session:::::APP_COMPANY_ID:<other-company-id>`) is used to attempt to overwrite
`APP_COMPANY_ID` (or `APP_USER_ID`/`APP_ROLE`) in session state, Then APEX rejects the attempt with a
Session State Protection error and every dashboard count remains unchanged from the pre-tamper
values — it never widens. **Build-verification prerequisite (part of this AC, not optional):** confirm
in App Builder that `APP_USER_ID`, `APP_COMPANY_ID`, and `APP_ROLE` all have **Session State
Protection = "Restricted — may not be set from browser"** (`sql/README.md` §2) before running the
tamper test — this manual, non-scriptable setting is the single point of failure for the entire
dashboard's isolation, since `V_MY_TICKETS` trusts these three items completely.

**AC-7 — Every dashboard region reads from the tenant-scoped view (enumerated checklist)** 🔒
Given each of the following regions, When its region SQL is inspected, Then it selects
`FROM V_MY_TICKETS` (never `FROM TICKETS` directly):
  - [ ] Status-count Card region (FR-17)
  - [ ] Priority-breakdown Chart series (FR-18)
  - [ ] Company-breakdown Chart series (FR-18)
  - [ ] Any later FR-28 KPI/analytics card (avg resolution time, per-agent counts — SHOULD, build after)

A region built against the base table is a defect regardless of what it displays correctly in testing
with the current seed data. **An authorization scheme that hides a region is not a substitute for that
region's source being `V_MY_TICKETS`** — this is the read-side analog of "hiding the button is not
sufficient" (cf. FR-10 AC-6).

**AC-8 — Company-name/legend lookup does not leak names of unseen companies** 🔒
Given the FR-18 company-breakdown chart needs a display name (not just `company_id`), When the label
lookup runs, Then it derives names only via a join against `COMPANIES` on the `company_id` values
**already present in the `V_MY_TICKETS`-scoped result set** — never a free-standing
`SELECT name FROM COMPANIES` that could populate a legend with companies the viewer can't otherwise
see. A Client Admin's chart must never show a Globex or Initech label, even greyed out or at zero.

**AC-9 — Breakdowns stay internally consistent per role**
Given any role, When the status breakdown (FR-17), priority breakdown, and company breakdown (FR-18)
are compared on the same page load, Then their totals agree (same underlying ticket set, just grouped
differently) — no region silently uses a different/unscoped source than its siblings (ties back to
AC-7's checklist).

**AC-10 — Drill-down never widens Page 4's source (conditional — only if a drill-down link exists)** 🔒
Given any Dashboard card/chart that links through to Page 4 (Ticket List/Queue) or a detail page, When
the link fires, Then it may only pre-filter/narrow (e.g. set a facet default such as status or
priority) — it must **never** supply a company predicate to Page 4's source, because Page 4 already
selects unconditionally `FROM V_MY_TICKETS`. Test: as a Client User or Client Admin, tamper the
incoming URL parameter to another tenant's company id → the resulting list is still empty or
own-company-only, never the tampered company's data. **This AC only applies if/when a drill-down link
is actually built** — it does not mandate building one; if no such link exists, this AC is trivially
satisfied and should be re-checked the moment one is added.

**AC-11 — Same page, four truths (the demo contrast)**
Given the Dashboard is one shared page — "one page serves many roles" (line 326), Page 3 marked
"Shared (role-filtered)" for all roles (line 342) — When each of the four seed accounts opens it in
turn, Then the page layout/charts are identical in structure but the **data** differs exactly per
AC-1–AC-4 — proving role-based access (non-negotiable #1) and multi-company isolation (non-negotiable
#2) live, on the dashboard (non-negotiable #4).

## Affected pages (of the 12, §6)

- **Page 3 — Dashboard** (Cards + Chart regions, MUST) — the sole surface for FR-17/18/19; every
  region built on `V_MY_TICKETS`. Owner P4.
- **Page 2 — Home/Landing** (MUST) — often redirects straight to the Dashboard; no scoping logic of
  its own, just routes to Page 3. Owner P4. Not an isolation surface — noted for completeness only.
- **Page 4 — Ticket List/Queue** (MUST) — downstream consumer if a drill-down link is added (AC-10);
  Page 4 already selects unconditionally `FROM V_MY_TICKETS` per its own build, so the only new risk
  is a leaked predicate arriving *from* Page 3's link, not Page 4's own source. Owner P3.

Data touched: read-only aggregates over `TICKETS` via `V_MY_TICKETS` (no writes) — all committed in
`sql/README.md`. No new columns or tables required.

## Demo path (the "same page, four truths" beat)

1. Log in as **sara@northwind.example** (System Admin) → open the **Dashboard** (Page 3) → point out
   status/priority/company charts covering **all three** client companies (AC-1).
2. Log out, log in as **aaron@acme.example** (Client Admin) → same Dashboard page, but every chart
   now shows **Acme only** — Globex/Initech absent (AC-2).
3. Log out, log in as **anna@acme.example** (Client User) → same page again, counts shrink further to
   **her own raised tickets** within Acme (AC-3) — the narrowest view of the four.
4. Log out, log in as **mike@northwind.example** (Support Agent) → Dashboard shows **Acme + Globex**
   combined, **zero Initech** (AC-4) — the vendor-side scoping contrast.
5. Call out explicitly: same page, same layout, four different truths — this single beat proves
   non-negotiables #1, #2 and #4 together (AC-11).

## Scope notes (no scope change)

- FR-19 as written traces cleanly to a committed page (3) and the committed isolation mechanism
  (`V_MY_TICKETS`, `sql/README.md`) — no new columns, tables, or brief change required.
- Companion MUSTs **FR-17** (counts by status) and **FR-18** (counts by priority/company) are the

- AC-10 (drill-down guardrail) is deliberately **conditional** — it does not mandate building a
  drill-down link from Dashboard to Queue; it exists so that *if* one is added (the mockup prototype
  already models this pattern in `docs/mockups/assets/app.js`), it's built isolation-safe from day one
  rather than retrofitted.

  MUST scoping demo: **FR-20** (bar/pie charts, SHOULD) and **FR-28** (avg resolution time + per-agent
  counts, SHOULD). Both are additive once AC-1–AC-11 demo clean; build them after, per §1 ("a broken
