Page 5 is the single page every role lands on after the queue — it's where **non-negotiable #1
(role-based access)** and **non-negotiable #2 (multiple companies)** get tested live and in
combination, not just at login. FR-9 has no edit semantics of its own; it's a **read surface over
three tenant-scoped views** (`V_MY_TICKETS`, `V_MY_COMMENTS`, `V_MY_HISTORY`) — get the read
scoping wrong here and every other FR that renders on this page (assign, status, comments,
escalate) inherits the leak.

## User stories

**Primary (FR-9):** As a **Client User**, I want to view the full detail of a ticket I raised —
subject, status, comment thread, and history — so that I can track progress without having to
ask support directly.

**Broader visibility:** As a **Client Admin**, I want to view the detail of any ticket raised by
my company (not just my own), so that I can oversee my team's open issues.

**Working context:** As a **Support Agent**, I want to view full ticket detail — including
internal notes — for tickets in the companies I cover, so that I have complete context before
acting on it.

**Full oversight:** As a **System Admin**, I want to view any ticket's detail across every
company, so that I can administer and audit anything in the system.

## Acceptance criteria (Given/When/Then)

**AC-1 — Happy path: Client User views own ticket**
Given I am a Client User and a ticket I raised exists, When I open it from the queue (Page 4) or
navigate to Page 5 with its `ticket_id`, Then I see subject, description, category, priority,
status, human-friendly reference (FR-8, e.g. `TKT-00003`), the comment thread (ordered oldest→newest
by `CREATED_AT`), and the history timeline — all read from `V_MY_TICKETS` / `V_MY_COMMENTS` /
`V_MY_HISTORY` (sql/05_isolation_views.sql lines 45–83).

**AC-2 — Client Admin sees any company ticket**
Given I am a Client Admin at Acme, When I open a ticket raised by another Acme user (not one I
raised myself), Then I see the same full detail — Client Admin visibility is "own company, all of
it" per the role table (brief §2, line ~106), distinct from Client User's additional
`created_by = me` restriction (see AC-11).

**AC-3 — Support Agent sees ticket detail scoped to covered companies** 🔒
Given I am a Support Agent covering Acme + Globex, When I open a ticket belonging to either
company, Then I see full detail including internal comments (agent-authored notes, FR-14). The
underlying view join is `AGENT_COMPANIES` (Decision I, `05_isolation_views.sql` lines 58–62), same
mechanism as the queue.

**AC-4 — System Admin sees any ticket, any company**
Given I am a System Admin, When I open any ticket regardless of company, Then I see full detail —
no `AGENT_COMPANIES` or `company_id` restriction applies to this role (§2; `05_isolation_views.sql`
lines 49–50, `WHEN V('APP_ROLE') = 'SYSTEM_ADMIN' THEN 1`).

**AC-5 — Internal comments hidden from clients (FR-14, SHOULD — already enforced by the view)** 🔒
Given a ticket has both client-visible and `is_internal = 'Y'` comments, When a Client User or
Client Admin views the comment region, Then internal comments are **absent**, not just visually
hidden — `V_MY_COMMENTS` filters them out server-side for client roles
(`05_isolation_views.sql` lines 70–75). Agents/System Admin see both.

**AC-6 — History region shows the full audit trail (FR-12)**
Given a ticket has had status changes, an assignment, and comments, When I view the History
region, Then every `TICKET_HISTORY` row appears in chronological order — region source is
`... ORDER BY CREATED_AT, HISTORY_ID` (the view itself carries no `ORDER BY`, per
`05_isolation_views.sql` lines 80–83, so the region query must add one, with `HISTORY_ID` as the
deterministic tiebreak for same-timestamp rows) — with `action` (fixed enum: `STATUS_CHANGE` /
`ASSIGN` / `ESCALATE` / `PRIORITY_CHANGE` / `COMMENT` / `CSAT`), `old_value`/`new_value`, actor,
and timestamp (brief §5 convention).

**AC-7 — Ticket reference is always displayed (FR-8)**
Given any ticket, When its detail page renders, Then the displayed reference is the human-friendly
`TKT-NNNNN` value read from `V_MY_TICKETS` — never the raw numeric `ticket_id`, and never re-derived
via a separate base-table lookup.

**AC-8 — Negative / cross-tenant IDOR on the ticket itself** 🔒
Given I am a Client User at Acme, When I edit the URL/page item to another company's `ticket_id`
(e.g. an Initech ticket), Then Page 5 shows **no data** (zero rows from `V_MY_TICKETS`), not an
error that confirms the ticket exists. The **view is the real backstop** — it fail-closes to zero
rows regardless of what the URL carries (`05_isolation_views.sql` lines 15–16, 45–64). Session
State Protection is **defense-in-depth**, not the primary guard: set `P5_TICKET_ID`'s SSP to
**"Checksum Required — Session Level"** (not `Restricted`, which is reserved for the server-stamped
`APP_*` items per sql/README §"2. Create 3 Application Items" — a `Restricted` PK item would make
it unsettable from the Page 4 link and break AC-1). The Page 4→5 navigation link is built via the
standard Page Designer link/branch, which auto-appends the `&cs=` checksum — no manual
`APEX_UTIL.PREPARE_URL` call is needed (reference/plsql/061-APEX_UTIL.md §61.98).

**AC-9 — Negative / cross-tenant IDOR on the embedded sub-regions** 🔒
Given the same tampered `ticket_id` scenario, When the Comments and History regions render
*independently* of the parent Form region, Then they **each** re-apply their own tenant filter —
region source is exactly `SELECT ... FROM V_MY_COMMENTS WHERE TICKET_ID = :P5_TICKET_ID` (resp.
`V_MY_HISTORY`) — **never** a join through the Form's already-fetched row, and never a base-table
query "for convenience." A sub-region must not trust that the page-level ticket lookup already
validated visibility, since each region issues its own query.

**AC-10 — Negative / cross-project block for Support Agent (Decision I)** 🔒
Given Mike (Support Agent, covers Acme + Globex only), When he attempts to open an Initech
ticket's detail (direct link or tampered PK), Then the page returns no data — `V_MY_TICKETS`
excludes it because `AGENT_COMPANIES` has no Mike→Initech row (`05_isolation_views.sql` lines
58–62). Lena (covers Globex + Initech) can open the same ticket. (Seed check per sql/README:
"Initech tickets must be invisible to Mike.")

**AC-11 — Negative / same-company, cross-user IDOR (Client User `created_by` predicate)** 🔒
Given I am **Anna** (Client User at Acme), When I tamper the `ticket_id` to a ticket raised by a
**different Acme user** (a coworker's ticket, not one I raised), Then Page 5 shows no data —
`V_MY_TICKETS` additionally requires `t.CREATED_BY = NV('APP_USER_ID')` for `CLIENT_USER`
(`05_isolation_views.sql` line 56), a predicate `CLIENT_ADMIN` does not carry. While Aaron
(Client Admin, same company) opens that identical ticket successfully (AC-2). This is the one AC
that exercises the branch separating Client User from Client Admin — distinct from the
cross-*company* cases in AC-8/AC-10.

**AC-12 — Output escaping on user-entered ticket text**
Given a ticket's `subject`, `description`, or any comment's `comment_text` contains HTML/script
characters (e.g. `<script>`), When the detail page renders any of these fields, Then the content
is displayed as literal text, not executed — column escaping stays ON (default) or explicit
`APEX_ESCAPE.HTML` is applied on any non-default render path (Cards, HTML Expression, custom
region template); no field uses an unescaped "HTML Expression" or raw Display-Only over these
columns (sql/README line 130; reference/plsql/027-APEX_ESCAPE.md).

**AC-13 — Detail Form region is read-only (no DML)** 🛠
Given the Page 5 Form region is sourced from `V_MY_TICKETS`, When the page renders, Then it
exposes **no** Automatic Row Processing (DML) — no Save/Apply Changes button wired to the view.
`V_MY_TICKETS` is not simply-updatable (`05_isolation_views.sql` lines 33–34); all writes
(assign FR-10, status/comment FR-11/FR-13, escalate FR-26) are separate guarded processes against
the base tables, each re-checking `SELECT COUNT(*) FROM V_MY_TICKETS WHERE TICKET_ID = :P_ID`
before writing (sql/README lines 115–121). This AC exists to stop a well-meaning "fix" — pointing
a default Form's DML at the base `TICKETS` table to make Save work — from silently reopening the
isolation hole this whole page was built to close.

## Demo path (the ticket-detail beat)

1. Log in as **anna@acme.example** (Client User) → open her ticket from the **Queue** (Page 4) →
   **Ticket Detail** (Page 5): reference, status, comment thread (no internal notes, chronological),
   full history (AC-1, AC-6, AC-7).
2. Log in as **mike@northwind.example** (Support Agent, covers Acme + Globex) → open the same
   ticket → sees an internal note anna doesn't (AC-3, AC-5).
3. **The isolation punch (three angles):**
   - Still as anna, edit the ticket-id in the URL to an **Initech** ticket (ivan@initech.example's)
     → Page 5 renders empty/blocked (AC-8).
   - Still as anna, edit the ticket-id to a **coworker's Acme** ticket (e.g. Aaron's) → also
     blocked, proving the block isn't just company-level (AC-11) — then log in as
     **aaron@acme.example** (Client Admin) and open that same ticket successfully.
   - As mike, attempt the Initech ticket → blocked (AC-10) — his queue never even lists it. Log in
     as **lena@northwind.example** (covers Globex + Initech) → same ticket opens fine, proving the
     block is project-scoped, not blanket.
4. Log in as **sara@northwind.example** (System Admin) → open any ticket from any company →
   full detail, full history, nothing hidden (AC-4).

Reinforces non-negotiable #1 (role-based) and #2 (multi-company) on the single page every role
converges on, and sets up the FR-10/FR-11/FR-13 demos that all render on this same hub.
