## User stories

**Primary (FR-4):** As a **Client User**, I want to see only tickets that belong to my own company,
so that I never see another client's confidential support data.

**Adjacent (§2, Client Admin row, line 106):** As a **Client Admin**, I want to see *all* tickets for my
own company (not just the ones I raised), so that I can oversee my company's support queue — while
still never seeing another company's tickets.

**Contrast case (§2, System Admin row, line 108):** As a **System Admin**, I want to see tickets
across every company, so that I can manage the whole desk — this is the designed exception, not a
leak, and proves the isolation logic branches on role rather than blindly filtering everyone the same way.

**Regression case (Decision I, FR-10 territory — not this ticket's scope, but shares the mechanism):**
A Support Agent's project-scoped view (`AGENT_COMPANIES`) reuses the same `V_MY_*` views this ticket
hardens — call it out here so a fix to FR-4 isolation is checked against the agent case too, but the
agent-scoping build-of-record lives in FR-10.

## Acceptance criteria (Given/When/Then)

**AC-1 — Client User sees only their own company's tickets (happy path)** 🔒
Given `anna@acme.example` (Client User, Acme Corp) logs in, When she opens the Ticket List/Queue
(Page 4), Then every row has `company_id` = Acme's, sourced `FROM V_MY_TICKETS` — never the
`TICKETS` base table (sql/README read rule, lines 112–113).

**AC-2 — Client Admin sees all of their own company, and only their own company** 🔒
Given `aaron@acme.example` (Client Admin, Acme Corp), When he opens the queue, Then he sees every
Acme ticket — including ones raised by `anna@acme.example` — but zero rows from Globex, Initech, or
Northwind (§2 line 106: "all tickets for their own company, never other companies'").

**AC-3 — Negative / cross-tenant list leak, client-side assertion** 🔒
Given `anna@acme.example` (Acme) or `george@globex.example` (Globex), When the ticket list renders,
Then no row belonging to `ivan@initech.example` (Initech), or to any company other than the viewer's
own, ever appears via `V_MY_TICKETS`. (This is distinct from the agent project-scoping seed test in
sql/README lines 34–35 — that proves Mike can't see Initech; this AC proves the *client* boundary
independently, since the two are different predicates in the same view.)

**AC-4 — Comments sub-region respects the same tenant boundary (happy path)** 🔒
Given a ticket Anna is allowed to see, When the Comments region (Page 5, FR-13) loads, Then it
selects `FROM V_MY_COMMENTS` — which both hides internal-only notes (FR-14) and enforces the same
`company_id` scoping as the ticket itself.

**AC-5 — History/audit timeline respects the same tenant boundary (happy path)** 🔒
Given a ticket Anna is allowed to see, When the History region (Page 5, FR-12) loads, Then it
selects `FROM V_MY_HISTORY` — same predicate, no base-table access.

**AC-6 — Negative / cross-tenant IDOR via tampered Ticket Detail URL or PK item** 🔒
Given `anna@acme.example`, When she edits the ticket PK in the URL (or via dev tools) to open a
Globex/Initech `ticket_id` on Ticket Detail (Page 5), Then **all three** regions — detail,
Comments (`V_MY_COMMENTS`), History (`V_MY_HISTORY`) — return zero rows together, and the page
performs a **branch/redirect to the Queue (Page 4)** with an explicit error region
("Ticket not found or you do not have access") rather than rendering a blank/broken detail page.
- **Primary, already-committed defense:** `V_MY_TICKETS`/`V_MY_COMMENTS`/`V_MY_HISTORY` return 0
  rows for a foreign PK regardless of any page-item control — this alone satisfies FR-4 (sql/README
  read rule). `TICKET_COMMENTS`/`TICKET_HISTORY` carry **no `company_id` column** (`sql/01_schema.sql`
  lines 119–150); isolation for these two tables depends **entirely** on the join inside the views,
  independent of whether the master ticket fetch succeeds — so this AC must be tested standalone,
  not inferred from the detail region failing.
- **Defense-in-depth, NEW/uncommitted work — route to 🛠 apex-expert:** set `P5_TICKET_ID`'s Session
  State Protection to **Checksum Required** (not *Restricted — may not be set from browser*: that
  level would break the legitimate Queue→Detail row link, which sets this item from the browser by
  design). Checksum Required still lets APEX build valid links while rejecting a hand-edited `ticket_id`.

**AC-7 — Write-path re-check, not just a hidden button (IDOR-on-write)** 🔒
Given any client-facing write against a ticket outside her company (e.g. a forged comment-insert
request bypassing a hidden UI element), When it is submitted, Then the process first runs
`SELECT COUNT(*) FROM V_MY_TICKETS WHERE TICKET_ID = :Pn` and rejects with
`raise_application_error` if the count is 0 (sql/README write rule, lines 118–121; brief §5 core
rule, line 272–273). Hiding the control is not sufficient — this is the same discipline the FR-10
self-assign ticket enforces for assignment.
> **Conditional watch-item, not in scope while Decision B is open:** *Reopen* (`Resolved → In
> Progress`, brief §4 line 213) is a client-triggered `STATUS_CHANGE` write on Page 5. **If** Decision
> B (§8, line 426) confirms Reopen ships in v1, its process needs this *exact same* `V_MY_TICKETS`
> write re-check before flipping status. Do not build this as a hard AC while B is undecided — track
> it here by name so it isn't missed if B lands "yes."

**AC-8 — App-item tamper resistance: `APP_COMPANY_ID` cannot be forged** 🔒
Given a logged-in Client User, When `APP_COMPANY_ID` is submitted or altered via a tampered URL or
POST, Then the value is rejected/unchanged — because the item's Session State Protection is
**Restricted — may not be set from browser** (sql/README lines 50–51, already committed) — and the
Queue/Dashboard/Detail continue to show only the original company's data. This is the keystone
control every other AC in this ticket implicitly relies on: every `V_MY_*` predicate reduces to
`WHERE company_id = NV('APP_COMPANY_ID')`, so if this one item were forgeable, every AC above would
be moot regardless of view correctness.

**AC-9 — Dashboard aggregates respect company scope (FR-19 tie-in)** 🔒
Given `anna@acme.example` or `aaron@acme.example`, When the Dashboard (Page 3, FR-17/FR-18/FR-19)
loads, Then every count/chart is computed only over Acme's tickets via `V_MY_TICKETS` — never a
global count leaking cross-tenant volume. This is FR-4's boundary applied to aggregates, feeding
judge non-negotiable #4 (dashboard) without breaking non-negotiable #2.

**AC-10 — Ticket creation stamps `company_id` server-side, never client-submitted** 🔒
Given `anna@acme.example` raises a ticket (Page 6, FR-7), When the insert commits, Then
`company_id = NV('APP_COMPANY_ID')` is stamped by the process — the page must not expose an
editable/submittable `company_id` item to Client User/Client Admin roles (sql/README insert rule,
line 123–125: "never map `COMPANY_ID` from a submittable page item").

**AC-11 — LOVs/pickers never leak across companies** 🔒
Given a client-facing form with a user picker (e.g. Client Admin raising a ticket on behalf of a
colleague, §2 line 106), When the LOV renders, Then it lists only users from her own company — never
a cross-company or global user list (sql/README LOV rule, line 127; `CATEGORIES` is the one
deliberate exception — it's global by design, not a leak, line 128).

**AC-12 — Export path inherits the same tenant scope** 🔒
Given the Ticket List/Queue's Interactive Report/Grid, When Anna downloads it (CSV/Excel, or any
future `APEX_DATA_EXPORT` path), Then the export runs against the same region source
(`V_MY_TICKETS`) — never a parallel query against the `TICKETS` base table. This is normally "free"
if the region itself is built correctly (the export always uses the region's own query), but must be
verified, not assumed, on Page 4.

**AC-13 — Correct exception: System Admin sees everything (not a leak)**
Given `sara@northwind.example` (System Admin), When she opens the queue or dashboard, Then she sees
tickets across all four companies — confirming the views branch on `APP_ROLE`, not a blanket
`company_id` filter that would (wrongly) also restrict her (§2 System Admin row, line 108).

**AC-14 — Companion control: no page/action leaks beyond what FR-3 allows**
Given `anna@acme.example` (Client User), When she navigates the app, Then she has no access to
Companies (Page 9) or Users (Page 10) admin grids — those are `IS_SYSTEM_ADMIN`-gated regardless of
company. This is FR-3's role gate, a companion requirement to FR-4, not the same control — listed
here for completeness, not as an FR-4 isolation surface.

## Demo path (the 30-second isolation "wow moment", brief §1 line 39)

1. Log in as **`anna@acme.example`** (Client User) → open the **Queue** (Page 4): only Acme tickets
   are listed. Open **Dashboard** (Page 3): counts are Acme-only (AC-1, AC-9).
2. Attempt to open a Globex/Initech ticket by editing the URL's ticket id → redirected to the Queue
   with "not authorized," detail/comments/history all empty together (AC-6).
3. Log out, log in as **`aaron@acme.example`** (Client Admin, Acme) → Queue now shows **all** Acme
   tickets, including Anna's — but still zero Globex/Initech/Northwind rows (AC-2, AC-3).
4. Log in as **`sara@northwind.example`** (System Admin) → Queue/Dashboard now show **every**
   company — the deliberate contrast that proves the branch is role-aware, not a blanket filter
   (AC-13).
5. *(Optional reinforcement, not this ticket's build)* Log in as **`mike@northwind.example`**
   (Support Agent, covers Acme+Globex) → confirm Initech is still invisible to him too, showing the
   same `V_MY_*` mechanism holds for the agent-scoping case owned by FR-10.
