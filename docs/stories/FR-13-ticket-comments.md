FR-13 isn't one of the four named judge non-negotiables, but it's the load-bearing content of
**Page 5 — Ticket Detail**, the hub every non-negotiable is demoed from, and it's the first place
role-based visibility (#1) gets tested on *data the client actually typed* — an internal agent
note is the one thing in this app that must be **invisible**, not just unauthorized. Get FR-13 +
its FR-14 gating wrong and the isolation story looks broken even if tickets/assignment are fine.

## User stories

**Primary (FR-13):** As a **Client User**, I want to add a comment to my own ticket, so that I can
give the support team more information or ask a follow-up question.

**Agent reply (FR-13):** As a **Support Agent**, I want to add a comment to a ticket in a company I
cover, so that I can communicate progress to the client while I work it.

**Internal note (FR-14, SHOULD, entangled with the same modal):** As a **Support Agent**, I want to
mark a comment as internal, so that I can leave notes for other agents/admins without the client
ever seeing them.

Supporting role: **Client Admin** — same capability as Client User, scoped to all of their
company's tickets (FR-2/role table, brief §2). **System Admin** — can comment on any ticket and,
being unrestricted, can also use the internal toggle.

## Acceptance criteria (Given/When/Then)

**AC-1 — Happy path: add a comment**
Given I can view a ticket (any role, via `V_MY_TICKETS`), When I open **Add Comment** (Page 8,
`P8_TICKET_ID` Value Protected, `P8_COMMENT_TEXT` Textarea Required) and submit non-blank text,
Then a `TICKET_COMMENTS` row is inserted with `TICKET_ID = :P8_TICKET_ID`, `COMMENT_TEXT`, `IS_INTERNAL`
per AC-3/AC-4, `CREATED_AT = SYSTIMESTAMP`, and `USER_ID` stamped server-side from `:APP_USER_ID` /
`NV('APP_USER_ID')` — a **Restricted** app item (sql/README.md line 51) — never bound from a
submittable page item, mirroring the `COMPANY_ID` stamp rule (sql/README.md lines 123–125). In the
same transaction `TICKETS.UPDATED_AT` is bumped (brief line 245, committed as an explicit
`UPDATE TICKETS SET UPDATED_AT = SYSTIMESTAMP WHERE TICKET_ID = :P8_TICKET_ID` step,
page-build-guide.md line 774). The comment appears in the Page 5 thread immediately, newest last.

**AC-2 — `TICKET_HISTORY` trace is deferred, not a build blocker**
A `TICKET_HISTORY` row with `action = 'COMMENT'` (fixed enum, brief line 246) is **not** required for
FR-13 to be demo-complete — it is explicitly deferred to Scope notes below rather than left as a
soft "may" AC (an untestable qualifier). Do not block sign-off on this row; if built, it follows the
same `INSERT ... VALUES (:P8_TICKET_ID, :APP_USER_ID, 'COMMENT', NULL, NULL)` pattern as the other
page processes.

**AC-3 — Internal toggle gated to agents/admins, enforced server-side (FR-14, SHOULD) 🔒**
Given I am a **Client User** or **Client Admin**, When I open Add Comment, Then the `P8_IS_INTERNAL`
Switch item is not rendered (Server-side Condition = `IS_AGENT` OR `IS_SYSTEM_ADMIN`,
page-build-guide.md line 762). This is **necessary but not sufficient**: the insert process itself
must re-derive the flag from role, not trust the item —
`CASE WHEN :APP_ROLE IN ('SUPPORT_AGENT','SYSTEM_ADMIN') THEN NVL(:P8_IS_INTERNAL,'N') ELSE 'N' END`
— so a forged POST of `P8_IS_INTERNAL=Y` from a Client User session is coerced to `'N'` at write
time, not merely hidden from the form. Conditional rendering alone does not satisfy this AC.

**AC-4 — Internal notes never reach the client (read isolation) 🔒**
Given a ticket has both a public reply and an internal note from an agent, When a **Client
User/Client Admin** views the Page 5 comment thread, Then the region's SQL is
`SELECT ... FROM V_MY_COMMENTS WHERE TICKET_ID = :P5_TICKET_ID` (sql/README.md line 108,
05_isolation_views.sql lines 70–75) — **never** the base `TICKET_COMMENTS` table with an inline
`IS_INTERNAL`/`:APP_ROLE` filter (see Scope note on doc drift below) — and only `IS_INTERNAL = 'N'`
rows render: the internal note is **absent**, not just styled differently. When a **Support
Agent/System Admin** views the same ticket, Then both rows render (`V_MY_COMMENTS` matches on
`V('APP_ROLE') IN ('SUPPORT_AGENT','SYSTEM_ADMIN')` — the actual stamped UPPER_SNAKE role values).

**AC-5 — Negative / cross-tenant IDOR on write 🔒**
Given a ticket belonging to a company I cannot see, When I submit Add Comment, Then the insert is
rejected — the process re-checks `SELECT COUNT(*) FROM V_MY_TICKETS WHERE TICKET_ID = :P8_TICKET_ID`
before writing (sql/README.md lines 118–121, bind variable — never string-concatenated). Two cases,
both must be named and tested:
  - **Client cross-company:** a Client User at Acme tampers `:P8_TICKET_ID` to a Globex ticket → rejected.
  - **Agent out-of-scope:** `mike@northwind.example` (covers Acme + Globex, **not** Initech) tampers
    `:P8_TICKET_ID` to an Initech ticket, with a public or internal comment → rejected by the same
    re-check (agent scoping via `AGENT_COMPANIES`, 05_isolation_views.sql lines 58–62). A hidden
    modal is not sufficient in either case.

**AC-6 — Negative / cross-tenant comment-thread leak 🔒**
Given the Page 5 comment region for any ticket, When it renders, Then it can never expose
`TICKET_COMMENTS` rows from a ticket belonging to another company — `V_MY_COMMENTS` carries the
same tenant predicate as `V_MY_TICKETS` (via the `TICKET_ID IN (SELECT TICKET_ID FROM V_MY_TICKETS)`
subquery), not a bare `WHERE ticket_id = :P5_TICKET_ID` against the base table.

**AC-7 — Validation**
Given the comment textarea is blank or whitespace-only, When I submit, Then the process rejects the
insert with an inline validation error and no `TICKET_COMMENTS`/`TICKET_HISTORY` row is written. The
`Required` attribute on `P8_COMMENT_TEXT` alone blocks empty submissions but **not**
whitespace-only text — add an explicit validation
(`WHEN TRIM(:P8_COMMENT_TEXT) IS NULL THEN 'Comment cannot be blank.'`), don't rely on `Required` alone.

**AC-8 — Output escaping on render (stored-XSS guard) 🔒 🛠**
Given `COMMENT_TEXT` is the one free-text column this entire ticket writes and re-renders, When the
Page 5 Conversation region displays it, Then it renders through the default auto-escaping Classic
Report column (**Escape special characters = Yes**, the APEX default) — no HTML expression / Cards
template bypasses this. Any custom-HTML render path instead wraps
`APEX_ESCAPE.HTML(c.COMMENT_TEXT)` (`reference/plsql/027-APEX_ESCAPE.md` §27.7, sql/README.md line
130). A stored payload in a comment (e.g. from a compromised/malicious client session) must not
execute when later viewed in an agent or System Admin session — this is a cross-tenant-adjacent
path (client-authored content executing in a higher-privilege session) even though the write itself
is bind-safe (page-build-guide.md line 777).

## Demo path

1. Log in as **anna@acme.example** (Client User) → open her Acme ticket (Page 5) → **Add Comment**
   (Page 8): no internal toggle visible (AC-3) → submit a public reply → it appears in the thread
   instantly (AC-1).
2. Log in as **mike@northwind.example** (Support Agent, covers Acme) → open the same ticket →
   Add Comment → check **Internal note** → submit → his note is visible to him, flagged internal.
3. Log back in as **anna@acme.example** → reopen the same ticket → confirm Mike's internal note is
   **absent** from her thread — only the public exchange shows (AC-4, the isolation punch).
4. As **anna@acme.example**, attempt to comment on a Globex ticket via a tampered `ticket_id` in
   the URL/modal PK → rejected (AC-5/AC-6) — reinforces non-negotiable #2 (multi-company) on the
   comment surface specifically, not just the ticket list.
5. As **mike@northwind.example**, attempt the same tamper against an **Initech** ticket (a company
   he doesn't cover) → rejected by the identical `V_MY_TICKETS` re-check (AC-5 agent case) —
   confirms `AGENT_COMPANIES` scoping extends to comments, not just assignment.
6. Log in as **sara@northwind.example** (System Admin) → same ticket → both public and internal
   comments visible, confirming the unrestricted admin view.

## Scope notes (no change needed)

- FR-13 as written traces cleanly to a committed page (8), a committed embedded region (Page 5),
  committed columns (`TICKET_COMMENTS`), and a defined isolation view (`V_MY_COMMENTS`). **No
  brief change required.**
- **Doc-drift flag for `/sync-docs`:** `docs/page-build-guide.md` lines 450–456 (Page 5 Conversation
  region) still shows the pre-view inline pattern —
  `FROM TICKET_COMMENTS c ... WHERE c.TICKET_ID = :P5_TICKET_ID AND (c.IS_INTERNAL = 'N' OR
  :APP_ROLE IN ('Support Agent','System Admin'))` — which is exactly the leaky base-table read
  `sql/README.md` line 112 forbids, superseded by `V_MY_COMMENTS`. Two problems with the stale
  snippet, both worth fixing in one pass: (1) it selects from the base table instead of the view,
  reopening the cross-tenant leak this ticket's AC-4/AC-6 close; (2) its role literals
  (`'Support Agent','System Admin'`) don't match the stamped `APP_ROLE` values, which are
  UPPER_SNAKE (`'SUPPORT_AGENT'`/`'SYSTEM_ADMIN'`, sql/README.md line 99) — as written the filter
  would hide internal notes from *everyone* (fails closed, but functionally broken, not just a
  style nit). The correct region SQL is
  `SELECT ... FROM V_MY_COMMENTS WHERE TICKET_ID = :P5_TICKET_ID ORDER BY CREATED_AT` — the view
  already encodes both the tenant scope and the correct-cased role check
  (05_isolation_views.sql lines 70–75). Recommend: `/sync-docs fix Page 5 Conversation region to
  read from V_MY_COMMENTS instead of the stale inline TICKET_COMMENTS query in page-build-guide.md`,
  then `python tools/sync_docs.py verify`.
- Watch-item, not a gap: the brief's own Page 8 description (line 348, a **MUST** page) already
  bakes in "internal-note flag for agents" — i.e. part of FR-14 (SHOULD) is structurally coupled
  to the MUST page. Recommend building the gated toggle (AC-3, including its server-side coercion)
  alongside AC-1 rather than literally "after all MUSTs," since it's the cheap half of FR-14 (one
  item + one authorization check) and it closes a client-visible-data isolation gap the moment
  comments exist. The *notification/analytics* trimmings of FR-14, if any emerge, can still wait —
  this does not reclassify FR-14 as a MUST, it only sequences its gating check early because
  Page 8 already ships it.
- The `TICKET_HISTORY` `'COMMENT'` row (brief line 246) is deferred out of the ACs (see AC-2) —
