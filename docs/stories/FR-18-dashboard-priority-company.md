**Judge non-negotiable #4 (Dashboard)** and **#2 (Multiple companies)** meet on this one FR. The
by-priority breakdown is a straightforward operational read; the **by-company** breakdown is the load-bearing
subtlety — for a **System Admin** it is the cross-company panorama that *is* the multi-tenant wow (#2),
but for a **Client Admin** it must collapse to their single company and for a **Client User** to just their
own tickets, with **no other tenant's name or count ever rendered**. Both breakdowns are counts over
`V_MY_TICKETS`, so role scoping (FR-19) and isolation are inherited from the view, not re-implemented per region.
Status counts (FR-17) are the sibling region on the same page; charts (FR-20) and per-agent/avg-resolution
analytics (FR-28) are adjacent SHOULDs and stay out of this MUST.

## User stories

**By priority (all roles):** As **any authenticated user**, I want the dashboard to show how many of the
tickets I can see fall under each priority (Low / Medium / High / Critical), so that I can gauge the
severity mix of my workload at a glance.

**By company (System Admin):** As a **System Admin**, I want the dashboard to show ticket counts broken
down per client company, so that I can see load across every tenant at once — the cross-company overview.

**By company (Client Admin):** As a **Client Admin**, I want the by-company breakdown to show **only my
own company**, so that I get a clean single-tenant total and never glimpse another client's name or volume.

**By company (Client User):** As a **Client User**, I want the same panels scoped to **only my own tickets**,
so that the dashboard reflects exactly what I am entitled to see.

## Acceptance criteria (Given/When/Then)

**AC-1 — By-priority counts (happy path)**
Given I am any authenticated user on Page 3, When the dashboard loads, Then a "By priority" region shows
one count per priority value — **Low / Medium / High / Critical** (§5) — each count = number of rows in
`V_MY_TICKETS` with that `priority`. Priorities with zero tickets show `0` (fixed four-bucket list, not
only-present values), so the mix is readable at a glance.

**AC-2 — By-company counts, System Admin (the multi-tenant view)** 🔒
Given I am a System Admin, When the dashboard loads, Then the "By company" region shows one count per
client company present in `V_MY_TICKETS` (which for this role spans all tenants) — e.g. Acme, Globex,
Initech side by side — each = the ticket count for that `company_id`. This is the cross-company panorama
(non-negotiable #2).

**AC-3 — By-company counts collapse for a Client Admin** 🔒
Given I am a Client Admin (e.g. aaron@acme.example), When the dashboard loads, Then the "By company"
region shows **exactly one row — my own company (Acme)** — because `V_MY_TICKETS` only returns my
company's rows. No other company name or count appears anywhere in the region.

**AC-4 — By-company counts scope to own tickets for a Client User** 🔒
Given I am a Client User (e.g. anna@acme.example), When the dashboard loads, Then both regions count only
the tickets `V_MY_TICKETS` exposes to me (my own raised tickets), and the by-company region shows only my
own company. No cross-user or cross-tenant totals are reachable.

**AC-5 — Counts are sourced from the isolation view, never base tables** 🔒
Given either region's query, When it runs, Then it selects `FROM V_MY_TICKETS` (never `FROM TICKETS`), so
the `WHERE company_id = :APP_COMPANY_ID` / role matrix is enforced once in the view. A count region that
reads a base table is a defect even if the number "looks right" for the current login.

**AC-6 — No tenant leak via the by-company breakdown** 🔒
Given a client-role session, When the by-company region renders, Then it can never surface a `company_name`
or aggregate count for a company other than the viewer's own — the company labels come only from rows
`V_MY_TICKETS` already returned, so an empty-for-me tenant is simply absent, not shown as `0`.

**AC-7 — Totals reconcile with the queue (consistency)**
Given the same login, When I compare the dashboard to Page 4 (Ticket List/Queue, FR-15), Then the sum of
the by-priority counts and the sum of the by-company counts each equal the total number of tickets that
queue shows me — both surfaces read the same scoped view, so they cannot disagree.

**AC-8 — Fail-closed outside a session** 🔒
Given the view's session functions resolve no `APP_COMPANY_ID` / `APP_ROLE` (no valid session), When a
region query executes, Then `V_MY_TICKETS` returns no rows and every count reads `0` — never a full-table
count (per the fail-closed guarantee in sql/README).

## Demo path (the 30-second dashboard beat)

1. Log in as **sara@northwind.example** (System Admin) → Page 3. Show **By priority** (Low/Medium/High/Critical)
   and **By company** listing **multiple tenants** (Acme, Globex, Initech) side by side — the cross-company
   overview (AC-1/AC-2, non-negotiable #2).
2. Log in as **aaron@acme.example** (Client Admin) → same Page 3. The **By company** region now shows
   **only Acme** — the panorama collapses to one tenant, no other client visible (AC-3). This is the
   isolation punch: same page, same code, different scope.
3. Log in as **anna@acme.example** (Client User) → counts shrink again to just her own tickets; still Acme-only
   (AC-4). Optionally cross-check her by-priority total against her Queue (Page 4) to show they reconcile (AC-7).
4. (Optional) **mike@northwind.example** (Support Agent) → by-company reflects only the companies he covers
   via `AGENT_COMPANIES` — same view, agent scope.

One page, three logins: "each client sees only their own, but we see everything" (§1, line 39) told entirely
through the dashboard.
